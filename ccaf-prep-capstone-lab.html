Before you start

You should be comfortable in a terminal and reading code — this lab assumes the developer experience the exam itself assumes (roughly 6+ months building with Claude). It does not stop to explain package managers or virtual environments.

  • Node 18+ or Python 3.10+ — the Agent SDK ships for both. Examples below are shown in both where it matters.
  • An Anthropic API key exported as ANTHROPIC_API_KEY. (The SDK also supports auth through Bedrock or Vertex if that’s your setup.)
  • A small Git repo with a real pull request or diff to point the agent at. Any repo you know well works; a repo with a deliberately-flawed branch works even better.
Verify before you build The Agent SDK, MCP tooling, and the structured-output API have all moved recently and continue to. Package names and a few flags below carry a real shelf-life. Wherever you see a Verify note, check it against the current docs at code.claude.com/docs and platform.claude.com/docs before relying on it. The shape of each pattern is stable; exact identifiers drift.
Orientation

What you’re building, before you build it

By the end of this lab you’ll have one working program: an agent that takes a pull request, reviews the changed code, produces categorized findings, and decides — per finding — whether to post a comment itself or hand it to a human. Every stage adds one capability to that same program. You are not building seven disconnected exercises; you are building one system in seven passes.

The one thing you’re building

A code-review agent. It reads a diff, reasons about it over several model turns, calls tools to fetch what it needs, extracts each problem as structured data, filters its own false positives, keeps its context small on big reviews, and escalates the calls it isn’t confident about.

input → a PR / diff output → categorized findings decision → post or escalate

How the seven stages connect

Read the diagram as a pipeline with a loop in the middle. The agent loop (Stage 1) is the engine; configuration (Stage 2) and tools (Stage 3) feed it; the loop’s output flows through extraction (Stage 4) and self-review (Stage 5); context management (Stage 6) wraps the loop to keep it healthy on long jobs; and escalation (Stage 7) is the final gate that decides where each finding goes.

Don’t worry if a box uses a word you don’t know yet — the glossary below defines every term in the diagram, and each stage explains its own box in full when you reach it.

The terms this lab uses

These come up across multiple stages. Skim them now; you don’t need to memorize them — they’ll make sense as you build.

Agent loop
The cycle the SDK runs for you: send the model a prompt, let it call a tool, run that tool, feed the result back, repeat — until the model produces a final answer instead of another tool call. The thing you’d otherwise hand-write with the raw Messages API.
Claude Agent SDK
The library that runs the agent loop, built-in tools, and context management for you, callable from Python or TypeScript. The renamed successor to the “Claude Code SDK” — old names claude-code-sdk / ClaudeCodeOptions are stale.
Tool
A capability the agent can invoke — read a file, run a search, fetch a diff. Built-in tools (Read, Grep, Glob, Bash…) ship with the SDK; you also define your own.
Subagent
A specialized agent the main agent spawns for a focused subtask. It runs in its own context and returns only its final message to the parent — used for specialization and to keep the parent’s context clean.
Session
One continuous run of the agent with memory of what happened. Each new call starts fresh unless you resume a prior session by its ID — that’s how the agent “remembers” an earlier review.
Context (window)
Everything the model can “see” at once — the conversation, files read, tool results so far. It’s finite, so a long review can fill it and degrade quality; managing it is Stage 6’s whole job.
CLAUDE.md
A project-context file the agent reads automatically. Layered by scope (user → project → subdirectory), with the more specific file winning.
Skill
A reusable capability packaged as a folder with a SKILL.md and YAML frontmatter (e.g. allowed-tools, context: fork). Here, a pr-review skill.
MCP (Model Context Protocol)
The open protocol that connects an agent to external systems — GitHub, a browser, a database — through servers that expose tools. How your agent reaches beyond the local files.
Structured output
Model output constrained to a JSON Schema you define, so it comes back in a fixed shape you can process. Guarantees the format, not the correctness — which is why validation still matters.
Message Batches API
An asynchronous endpoint for bulk requests — up to 100k per batch, results within 24 hours, at half the token cost. For processing a queue of PRs, not interactive review.
Escalation / human-in-the-loop
Handing a decision to a person instead of resolving it autonomously — on a defined trigger (policy gap, explicit request, can’t progress) or when confidence is too low to act alone.
Finding
One issue the review produces — a file, an optional line, a severity, and a description. The unit of data that flows through extraction, review, and the confidence gate.
Before Stage 1
  • You can state, in one sentence, what the finished agent does end to end.
  • You can trace a single finding through the diagram: loop → extract → review → gate.
  • The terms in the glossary look familiar — you’ll cement them by building.
Stage 01

Stand up the agent loop

Agentic Architecture & Orchestration · 27%

The whole project rests on one thing: a loop that calls the model, runs the tools it asks for, feeds results back, and repeats until the work is done. Build that first, get it resuming across turns, then split off a subagent.

A pull request lands. Your agent needs to read the changed files, reason about them, and produce findings — possibly over many model turns and tool calls.

1a · Install and make one query

The Claude Agent SDK is the renamed successor to the old “Claude Code SDK.” If a tutorial imports claude-code-sdk or ClaudeCodeOptions, it’s stale — the code won’t run.

Install
# Python pip install claude-agent-sdk # TypeScript (bundles the Claude Code binary — nothing else to install) npm install @anthropic-ai/claude-agent-sdk
A first query (Python)
import asyncio from claude_agent_sdk import query, ClaudeAgentOptions async def main(): async for message in query( prompt="Read the diff in ./pr and list the files it touches.", options=ClaudeAgentOptions( allowed_tools=["Read", "Grep", "Glob"], permission_mode="acceptEdits", # or gate with canUseTool ), ): if hasattr(message, "result"): print(message.result) asyncio.run(main())

You did not write the loop. query() runs the full model–tool–result cycle for you and streams every intermediate message back, ending with a result message. That’s the point of the SDK over the raw Messages API — the stop_reason == "tool_use" plumbing is handled.

1b · Handle errors and cap the run

The SDK does not ship default turn limits, budget caps, or retry logic. An unbounded agent can loop and burn tokens with no circuit breaker. Set limits explicitly.

options=ClaudeAgentOptions( allowed_tools=["Read", "Grep", "Glob"], max_turns=12, # circuit breaker: don't run forever # max_budget_usd is an optional cap — set it in production )

Wrap the loop in your own try/except and inspect the result message’s error subtype. Model-call failures surface discriminated codes such as rate_limit, overloaded, authentication_failed, and server_error — branch your retry logic on those rather than treating every failure the same.

1c · Keep context across turns (sessions)

Every query() starts a fresh session with no memory of the last one. To continue — e.g. review, then answer a follow-up about the same PR — capture the session_id from the init message and pass resume on the next call.

# First call: grab session_id from the init message. # Second call: resume it. options=ClaudeAgentOptions(resume=session_id)

1d · Split off a reviewer subagent

Define a specialized agent in the agents parameter, and include Agent in allowedTools so the parent is allowed to spawn it. The subagent runs in its own context — it can read fifty files without any of that noise landing in the parent’s conversation. Only its final message returns to the parent.

TypeScript
import { query } from '@anthropic-ai/claude-agent-sdk'; const result = query({ prompt: "Review the auth module changes for security issues.", options: { allowedTools: ["Read", "Grep", "Agent"], // Agent = allowed to delegate agents: { 'security-reviewer': { description: 'Reviews code for security flaws. Use for auth, secrets, input handling.', prompt: 'You are a security review specialist. Report only exploitable issues, with severity.', tools: ["Read", "Grep"], // read-only: it can analyze, never modify }, }, }, });
Exam lens Scenarios here ask when to spawn a subagent versus work in the main context. The right answer is almost always about context isolation or parallelism — not “subagents are better.” For a single sequential read, delegating just adds overhead. “Subagent sprawl” (spawning too many) is a real failure mode the exam likes to test.
Verify Package names, ClaudeAgentOptions fields, and the exact session-resume shape are current as of mid-2026 but the SDK is pre-1.0 and moving (there’s already a V2 session interface alongside the generator pattern). Confirm the import path and options object against code.claude.com/docs/en/agent-sdk.
Stage 1 done when
  • Your agent reads a real diff and returns findings over multiple turns.
  • A follow-up question resumes the same session with full context.
  • One read-only reviewer subagent runs and returns a summary to the parent.
  • max_turns is set, and you handle at least one error subtype deliberately.
Stage 02

Configure Claude Code around the repo

Claude Code Configuration & Workflows · 20%

Your agent is only as good as the project context it reads. Set up the configuration hierarchy so the agent knows your conventions, restrict what it can do in sensitive paths, package a reusable review capability as a skill, and wire in one MCP server.

2a · The CLAUDE.md hierarchy

The SDK reads CLAUDE.md from the working directory as persistent project context, the same file Claude Code uses. Precedence runs from broad to specific — more specific wins:

  • User — ~/.claude/CLAUDE.md: your personal defaults across every project.
  • Project — ./CLAUDE.md at the repo root: shared, committed to version control.
  • Subdirectory — a CLAUDE.md deeper in the tree refines the rules for that area.

Add a .claudeignore to keep the agent out of build artifacts, vendored code, and secrets.

./CLAUDE.md
# Code Review Agent — project context Review Python and TypeScript changes only. Flag: unhandled errors, missing input validation, hardcoded secrets. Do NOT comment on formatting — a linter handles that. Severity scale: blocker > major > minor > nit.

2b · Path-specific rules

Put narrower rules in .claude/rules/ so behavior can differ by area of the codebase — e.g. stricter handling under infra/ or migrations/ than under docs/.

2c · Build a review skill

A skill is a folder with a SKILL.md and YAML frontmatter. Two frontmatter options matter for this build:

  • allowed-tools — restricts what the skill may do (a review skill reads, it doesn’t write).
  • context: fork — runs the skill in an isolated subagent so its verbose output doesn’t pollute your main session context. This one shows up on the exam by name.
.claude/skills/pr-review/SKILL.md
--- name: pr-review description: Review a pull request diff and produce categorized findings. allowed-tools: Read, Grep, Glob context: fork --- # PR Review Read the diff, group findings by severity, and return one line per finding: severity | file:line | what | why.

2d · Integrate one MCP server

MCP is how the agent talks to external systems. Config location signals scope: .mcp.json at the project root is shared through version control; ~/.claude.json is your personal, experimental config. Wire in at least one server — a GitHub MCP server to fetch PRs, or the Playwright server if the change touches a UI.

# In the SDK, MCP servers are passed inline: mcp_servers={ "playwright": {"command": "npx", "args": ["@playwright/mcp@latest"]} }
Exam lens Expect questions on precedence (which CLAUDE.md wins) and scope (.mcp.json shared vs. ~/.claude.json personal). And know what context: fork buys you: isolation, so a noisy skill doesn’t bloat the caller’s context. Distractors will offer plausible-but-wrong frontmatter keys.
Verify Skill frontmatter keys and the MCP config filenames are current, but MCP server packages and the scaffolding CLI change often. Confirm context: fork and allowed-tools spelling, and the server’s install command, against the live Claude Code docs before committing them.
Stage 2 done when
  • A project CLAUDE.md steers the agent’s review, and a subdirectory rule overrides it somewhere.
  • Your pr-review skill runs with context: fork and read-only tools.
  • One MCP server is connected and the agent uses it during a review.
Stage 03

Design & test the MCP tools

Tool Design & MCP Integration · 18%

The agent will only pick the right tool if your descriptions make the choice obvious. Two similar tools with vague descriptions is the single most common cause of misrouting. Write them to differentiate, return errors the agent can act on, then deliberately try to confuse it.

3a · Differentiate similar tools

Give your review agent two tools that are easy to confuse: fetch_pr_diff and fetch_file_contents. Write descriptions that draw a sharp boundary — state what each is for and, crucially, when not to use it.

{ "name": "fetch_pr_diff", "description": "Get ONLY the changed lines in a pull request, as a unified diff. " "Use to review what a PR changes. Do NOT use to read a whole file — " "it returns diffs, not full contents." } { "name": "fetch_file_contents", "description": "Get the COMPLETE current contents of one file at a path. " "Use when you need surrounding context beyond the diff. " "Do NOT use to see what changed — use fetch_pr_diff for that." }

3b · Structured error responses

When a tool fails, return an error the agent can reason about — a category and a retryable flag — not a bare string. This is what lets the loop decide whether to retry, fall back, or give up.

{ "error": { "category": "rate_limited", // not_found | invalid_input | rate_limited | upstream_down "retryable": true, "retry_after_s": 30, "message": "GitHub API rate limit hit. Retry after 30s." } }

Categories let the agent branch: invalid_input means fix the arguments and retry; not_found means stop and report; rate_limited means wait; upstream_down means fall back or escalate.

3c · Test selection under ambiguity

Now try to break it. Feed the agent requests that could route to either tool and see which it picks:

  • “Show me what changed in auth.py” → should pick fetch_pr_diff.
  • “I need to see the whole auth.py to understand the change” → should pick fetch_file_contents.
  • “Look at auth.py” → genuinely ambiguous. A well-described tool set makes the agent ask or pick defensibly; a poorly-described one coin-flips.

If the agent routes wrong, the fix is almost always the description, not the model. Tighten the “do NOT use when” clause and re-test.

Exam lens This domain is judgment about tool boundaries. Questions show two overlapping tools and ask why the model mis-selects, or which description fixes it. The winning answer sharpens the boundary and adds negative guidance (“do not use for X”). Structured errors with categories and retryable flags are the other reliable theme — know why a category beats a string.
Stage 3 done when
  • Your two tools have descriptions with explicit “do NOT use when” boundaries.
  • Tool errors return a category + retryable flag, and the loop branches on them.
  • You’ve run at least three ambiguous requests and logged which tool got picked.
Stage 04

Extract findings as structured data

Prompt Engineering & Structured Output · 20%

Free-form review text is unusable downstream. Turn each finding into a validated JSON object so you can post comments, count blockers, and gate a merge. Then design the schema so it survives messy real-world input, and batch the whole thing for a large PR queue.

4a · Constrain output to a schema

Structured outputs are now generally available on the current Claude models via output_config.format, which compiles your JSON Schema into a grammar and constrains generation so the model literally cannot emit non-conforming tokens. Define the shape of a finding:

finding_schema = { "type": "object", "additionalProperties": False, "properties": { "file": {"type": "string"}, "line": {"type": ["integer", "null"]}, # nullable: some findings are file-level "severity": {"type": "string", "enum": ["blocker","major","minor","nit"]}, "issue": {"type": "string"}, "suggestion": {"type": "string"} # optional, see 4c }, "required": ["file", "severity", "issue"] }
Python — the GA path
message = client.messages.create( model="claude-sonnet-4-6", max_tokens=1024, messages=[{"role": "user", "content": diff_text}], output_config={"format": {"type": "json_schema", "schema": findings_schema}}, )
An older pattern defines the schema as a tool and forces it with tool_choice. That still works and is still needed for some cases (older models, extended-thinking combinations), so recognize it — but prefer output_config.format when the model supports it.

4b · Validation-retry loop

Constrained decoding guarantees the shape, not the content — you can still get a perfectly-formatted wrong answer. Validate against your own rules (does the file exist in the diff? is the line in range?) and retry with the validation error fed back in:

for attempt in range(3): result = extract(diff_text, prior_error=err) err = validate(result) # your semantic checks if err is None: break # feed err back into the next prompt

4c · Design for missing data

Real diffs are messy. Make fields the model genuinely can’t always fill optional (leave them out of required) or nullable (["integer","null"]). Forcing a required line on a file-level finding pushes the model to invent one — the opposite of what you want.

4d · Batch a PR queue

For a backlog of PRs, don’t call synchronously in a loop. The Message Batches API takes up to 100,000 requests per batch, returns within 24 hours, and costs 50% less on input and output tokens. Tool definitions and structured-output schemas are accepted per-request inside the batch, so each PR gets the same schema-constrained extraction.

Exam lens Two favorites here: optional/nullable schema design (forcing a required field the model can’t know invites fabrication) and when to batch (async, non-real-time, cost-sensitive bulk — not interactive review). Also know that structured outputs guarantee format, not correctness — hence the validation-retry loop still earns its place.
Verify Structured outputs graduated from a beta header (structured-outputs-2025-11-13) to GA output_config.format; which models support the GA path changes as new models ship. Confirm your target model and the exact field name against platform.claude.com/docs/en/build-with-claude/structured-outputs.
Stage 4 done when
  • Findings come back as schema-valid JSON, with at least one nullable and one optional field used on purpose.
  • A validation-retry loop catches a semantic error and re-prompts.
  • You’ve run a small batch of PRs through the Batches API with the same schema.
Stage 05

Add the self-review pass

Prompt Engineering & Structured Output · 20%

A first-pass reviewer over-flags. It calls style preferences “blockers” and invents problems in code it half-understands. Cut the false positives with worked examples, explicit criteria, and a second pass that reviews the review.

5a · Few-shot the ambiguous calls

The hard cases aren’t obvious bugs — they’re judgment calls. Show the agent 2–3 worked examples of borderline findings and the correct severity, so it calibrates instead of guessing.

# In the review prompt: Example — a broad `except:` in a one-off migration script → severity: minor (acceptable in throwaway code; would be major in a request handler). Example — a hardcoded API key in any committed file → severity: blocker, always. Example — a variable named `data` → not a finding. Do not report naming taste.

5b · Explicit review criteria

Vague instructions (“find problems”) produce noise. Define what counts as reportable and what explicitly does not, so the agent stops surfacing non-issues:

  • Report: anything that can cause incorrect behavior, a security hole, or data loss.
  • Do not report: style, formatting, naming taste, or anything a linter already catches.
  • When unsure whether something is exploitable: mark it minor and say what you’re unsure about — don’t inflate it to blocker.

5c · Multi-pass architecture for big reviews

A large PR won’t review well in one pass. Structure it as stages, each with a narrow job:

  1. Pass 1 — collect: each file (or subagent) produces raw findings independently.
  2. Pass 2 — filter: a reviewer pass drops false positives against the criteria from 5b.
  3. Pass 3 — rank: surviving findings are deduped and ordered by severity for the human.

This is where the parallel subagents from Stage 1 pay off: run a style-checker, a security-scanner, and a test-coverage reviewer concurrently, then merge. Each keeps its own context, so one file’s deep read doesn’t crowd out another’s.

Exam lens The through-line is reducing false positives. Expect scenarios where an agent over-flags and you choose the fix: few-shot examples for the ambiguous band, explicit exclusion criteria, or a filtering pass. “Add more instructions telling it to be careful” is the tempting wrong answer; concrete examples and criteria win.
Stage 5 done when
  • Your prompt carries 2–3 few-shot examples covering borderline severity calls.
  • Explicit report / do-not-report criteria measurably cut the noise on a real PR.
  • A second pass filters the first pass’s output before anything reaches a human.
Stage 06

Manage context on long reviews

Context Management & Reliability · 15%

A big PR, a long thread, verbose tool output — the context window fills and quality drifts. Keep the working context small and well-organized: extract facts from noise, write a scratchpad, and delegate to keep the parent lean.

6a · Extract facts from verbose tool output

A raw diff or test log can be thousands of tokens of mostly-irrelevant text. Don’t feed all of it forward. Extract the structured facts the agent actually needs — changed symbols, failing tests, touched public APIs — and carry those instead of the raw dump.

# Instead of carrying 4,000 tokens of test output, carry: { "failing_tests": ["test_auth_expiry"], "changed_public_fns": ["validate_token"] }

6b · Scratchpad for long sessions

For a review that spans many turns, write intermediate state to a file rather than holding it all in context. A running findings.md or a task list on disk means the agent — or a fresh session — can pick up where it left off without re-reading everything. The pattern generalizes: state lives outside the context window; the window holds only what the current step needs.

6c · Delegate to stay under the limit

This is the second reason subagents exist (the first was specialization). A research subagent can read twenty files and return a three-line summary — the parent pays for the summary, not the twenty files. On a large review, delegating per-file or per-concern is how you keep the orchestrator’s context from ballooning.

Watch the orchestrator’s cost. Every parent turn carries the full conversation plus all subagent descriptions, so long sessions get progressively more expensive. The SDK compacts automatically as the limit approaches and re-reads CLAUDE.md after compaction — but designing for small context beats relying on compaction.
Exam lens Scenarios describe an agent degrading over a long session and ask for the fix. The answers cluster around fact extraction (carry the distilled facts, not the raw output), external state (scratchpad/files over in-context memory), and delegation for context isolation. Know why each shrinks the working context.
Stage 6 done when
  • Verbose tool output is distilled to structured facts before moving forward.
  • A scratchpad file holds review state across turns / a resumed session.
  • Delegation keeps the parent’s context small on a genuinely large review.
Stage 07

Route low-confidence findings to a human

Context Management & Reliability · 15%

The last piece is knowing what not to decide alone. A production review agent resolves what it’s sure about and escalates the rest — on a clear trigger, through a real workflow, routed by confidence.

7a · When to escalate vs. resolve

Escalate on a defined trigger, not a vibe. The three that matter here:

  • Policy gap — the change touches something no rule covers (a new dependency’s license, a data-handling question). The agent has no basis to decide.
  • Explicit request — the PR author or a rule asks for human sign-off (e.g. anything under migrations/).
  • Inability to progress — the agent can’t reach a confident conclusion after its passes: conflicting signals, or a finding it can’t verify is exploitable.

Everything else — the clear blockers, the obvious nits — it resolves autonomously.

7b · Confidence-based routing

Have the review attach a confidence to each finding, then route on it. A simple, defensible policy:

if finding.confidence >= 0.85: auto_post(finding) # high confidence → post the comment elif finding.confidence >= 0.5: queue_for_human(finding) # uncertain → human review queue else: drop_or_flag_as_low_signal(finding) # too weak → don't waste a human on it

The band in the middle is the point: the agent is neither confident enough to act nor confident enough to discard, so a person decides. That’s the human-in-the-loop workflow — not “a human checks everything,” which defeats the purpose, and not “the agent decides everything,” which is how bad calls ship.

7c · Design the review workflow

Make escalation a real path, not a dead end: a queue the human can see, enough context attached for them to decide fast (the finding, the diff hunk, why the agent was unsure), and a way for their decision to flow back — ideally captured so tomorrow’s policy gap becomes next week’s rule.

Exam lens This is pure judgment, and it’s scenario-heavy. Questions give a situation and ask whether the agent should escalate or resolve. Map to the triggers: policy gap, customer/author request, can’t progress → escalate; clear and within policy → resolve. Confidence-based routing (auto / human / drop) is the canonical design. Beware answers that route everything to a human or nothing — both miss the point.
Stage 7 done when
  • Your agent escalates on at least one real trigger and resolves clear cases itself.
  • Findings carry a confidence score that routes them to auto-post, human, or drop.
  • Escalations land in a queue with enough context for a human to decide quickly.
Wrap-up

What you built, and how it maps

You now have one working agent that exercises all five exam domains: an agent loop with sessions and subagents (27%), a Claude Code configuration around it (20%), differentiated MCP tools with structured errors (18%), a schema-constrained extraction pipeline with a self-review pass (20% + 20%), and context management with confidence-based escalation (15%). That’s not coverage by coincidence — it’s the same production system the scenario questions are drawn from.

The exam is 60 scenario questions in 120 minutes, and it rewards judgment over recall — the wrong answers are written to tempt people with partial knowledge. Having actually built each stage is the difference between recognizing the right architectural call and guessing at it.
