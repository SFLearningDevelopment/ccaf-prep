Section 1.1

Agentic loops for autonomous task execution

Task Statement 1.1

An agent isn’t a single call — it’s a loop. Claude decides it needs a tool, you run the tool, you feed the result back, and Claude decides what to do next. Getting the loop’s control flow right — and knowing what not to base it on — is the foundation the whole domain rests on.

What & why

The agentic loop is the cycle that lets Claude execute a multi-step task on its own. Each turn, you send the conversation to Claude and inspect the response’s stop_reason. If it’s "tool_use", Claude is asking you to run a tool — you execute it, append the result to the conversation, and loop again. If it’s "end_turn", Claude is done — you stop. That single signal, stop_reason, is the correct control-flow mechanism, and building the loop around anything else is where most agent bugs come from.

Knowledge points
  • The loop lifecycle: send request → inspect stop_reason ("tool_use" vs "end_turn") → execute requested tools → return results for the next iteration.
  • Tool results are appended to conversation history so the model can reason about the next action with the new information in context.
  • The distinction between model-driven decision-making (Claude reasons about which tool to call next based on context) and pre-configured decision trees or fixed tool sequences. The loop is driven by the model, not a hardcoded flowchart.
In practice
  • Continue the loop while stop_reason == "tool_use"; terminate when it’s "end_turn". That’s the whole control condition.
  • After executing each tool, add a tool_result block to the conversation before the next request, so the model incorporates it into its reasoning.
  • Let the model choose the next tool from context rather than encoding a fixed sequence — that adaptability is the point of an agent.
Anti-patterns the exam tests
  • Parsing natural-language signals to decide the loop is done (“if the reply says ‘finished’…”) — brittle and wrong; use stop_reason.
  • Arbitrary iteration caps as the primary stopping mechanism — a safety backstop is fine, but the loop should terminate on end_turn, not because it hit turn 10.
  • Checking for assistant text content as a completion indicator — a response can contain text and a tool call; text presence doesn’t mean done.
Exam lens A scenario shows an agent that stops too early or loops forever. The tempting answers add a turn cap or scan the text for a keyword. The right answer is almost always “drive termination off stop_reason” — end_turn stops, tool_use continues. If an answer relies on reading the model’s prose to decide control flow, it’s the distractor.
⚑
Build it in a capstone Every SDK lab runs this loop, but the Customer Support Resolution Agent lab makes the loop’s control flow explicit — watch where stop_reason drives the intake→lookup→resolve cycle.
Check your understanding
  • Can you state the exact condition that continues the loop, and the one that ends it?
  • Can you name three things that must not be used to decide the loop is complete?
  • Why must tool results be appended to history before the next request?
Section 1.2

Orchestrate multi-agent systems with coordinator–subagent patterns

Task Statement 1.2

One agent handles one task well; hard problems need several, coordinated. The dominant pattern is hub-and-spoke: a coordinator that decomposes the work, delegates to specialist subagents, and aggregates what comes back — routing all communication through itself.

What & why

In a coordinator–subagent (hub-and-spoke) architecture, a single coordinator manages all inter-subagent communication, error handling, and information routing. Subagents don’t talk to each other; they talk to the coordinator. This centralization is what gives the system observability, consistent error handling, and controlled information flow — the coordinator is the one place you can see and govern what’s happening. Its job is task decomposition, delegation, result aggregation, and deciding which subagents to invoke based on the query, rather than always running the full pipeline.

Knowledge points
  • Hub-and-spoke: the coordinator manages all inter-subagent communication, error handling, and routing — subagents don’t communicate directly.
  • Subagents operate with isolated context — they do not automatically inherit the coordinator’s conversation history.
  • The coordinator’s role: decomposition, delegation, aggregation, and deciding which subagents to invoke based on query complexity.
  • The risk of overly narrow decomposition — partitioning a broad research topic so tightly that coverage ends up incomplete.
In practice
  • Design coordinators that analyze the query and dynamically select which subagents to invoke, rather than always routing through the whole pipeline.
  • Partition scope to minimize duplication — assign distinct subtopics or source types to each subagent.
  • Implement iterative refinement: the coordinator evaluates synthesis output for gaps, re-delegates targeted queries to search/analysis subagents, and re-invokes synthesis until coverage is sufficient.
  • Route all subagent communication through the coordinator for observability and consistent error handling.
Anti-patterns the exam tests
  • Always running the full pipeline regardless of the query — a simple question shouldn’t fan out to every subagent; the coordinator should select.
  • Subagents communicating directly with each other — breaks observability and centralized error handling; everything routes through the hub.
  • Over-narrow decomposition that leaves a broad topic with gaps — the coordinator must check coverage and re-delegate, not assume the first partition was complete.
  • Assuming subagents inherit context — they start fresh; anything they need must be passed explicitly (see 1.3).
Exam lens A scenario describes a research system missing whole subtopics, or two subagents duplicating work. The fix isn’t “add more subagents” — it’s the coordinator partitioning scope to avoid overlap and running an iterative refinement loop that detects gaps and re-delegates. Answers where subagents coordinate among themselves, or where the pipeline runs identically for every query, are the distractors.

Check your understanding
  • Why does all subagent communication route through the coordinator rather than agent-to-agent?
  • What does “dynamic selection” buy you over always running the full pipeline?

Section 1.3

Configure subagent invocation, context passing, and spawning

Task Statement 1.3

If 1.2 is the shape of a multi-agent system, this is the wiring: how a coordinator actually spawns a subagent, what crosses the boundary between them, and how to run several at once. The recurring truth here — a subagent knows only what you put in its prompt.

What & why

Subagents are spawned through the Task tool, and a coordinator can only invoke them if "Task" is in its allowedTools — miss that, and delegation silently collapses to a single agent doing everything itself. Because subagents run in isolated context, everything they need must be provided explicitly in the prompt: they don’t inherit parent context or share memory between invocations. Each subagent type is described by an AgentDefinition — its description, system prompt, and tool restrictions.

Knowledge points
  • The Task tool is the spawning mechanism; allowedTools must include "Task" for a coordinator to invoke subagents.
  • Subagent context must be explicitly provided in the prompt — no automatic inheritance, no shared memory across invocations.
  • AgentDefinition configures each subagent type: description, system prompt, tool restrictions.
  • Fork-based session management for exploring divergent approaches from a shared analysis baseline (see also 1.7).
In practice
  • Pass complete findings from prior agents directly in the subagent’s prompt — e.g., feed web-search results and document-analysis output into the synthesis subagent’s prompt.
  • Use structured data formats to separate content from metadata (source URLs, document names, page numbers) so attribution survives the handoff.
  • Spawn parallel subagents by emitting multiple Task calls in a single coordinator response — not across separate turns.
  • Write coordinator prompts that specify goals and quality criteria, not step-by-step procedures, so subagents can adapt.
Anti-patterns the exam tests
  • Assuming a subagent can see the coordinator’s conversation — it can’t; unpassed context is invisible to it.
  • Forgetting "Task" in allowedTools — the coordinator then can’t delegate and quietly runs as one agent.
  • Spawning parallel work across separate turns instead of multiple Task calls in one response — serializes what should be concurrent.
  • Passing procedural step-by-step instructions that prevent the subagent from adapting to what it finds.
  • Compressing context so metadata (source, page, date) is stripped before the handoff, destroying downstream attribution.
Exam lens A scenario shows a synthesis agent producing uncited or context-free output. The tempting answer tweaks the synthesis prompt’s wording; the right answer is that the coordinator must pass the prior findings (with their source metadata) explicitly into the synthesis prompt — the subagent never had them otherwise. Any answer assuming shared memory between agents is wrong by construction.
⚑
Build it in a capstone The Multi-Agent Research System lab passes each subagent’s findings forward in the next prompt and preserves source metadata through synthesis — this section’s skills, made concrete.
Check your understanding
  • What single allowedTools entry makes delegation possible, and what happens without it?
  • How do you spawn subagents in parallel rather than serially?
  • Why must source metadata travel as structured data across the handoff?
Section 1.4

Implement multi-step workflows with enforcement and handoff patterns

Task Statement 1.4

Some ordering is a preference; some is a hard requirement. When a step must happen before another — verify identity before moving money — a prompt instruction isn’t enough, because prompts have a non-zero failure rate. That’s where programmatic enforcement comes in.

What & why

There are two ways to order a workflow: programmatic enforcement (hooks, prerequisite gates that block a tool until a condition is met) and prompt-based guidance (telling the model the order). They are not interchangeable. When deterministic compliance is required — identity verification before a financial operation — prompt instructions alone will eventually fail, because the model follows them probabilistically. For those cases you enforce the prerequisite in code. This section also covers structured handoffs: when a process escalates mid-stream, the human receiving it needs customer details, root-cause analysis, and recommended actions in a compiled summary.

Knowledge points
  • The difference between programmatic enforcement (hooks, prerequisite gates) and prompt-based guidance for workflow ordering.
  • When deterministic compliance is required, prompt instructions alone have a non-zero failure rate — not good enough for identity-before-refund.
  • Structured handoff protocols for mid-process escalation: customer details, root-cause analysis, recommended actions.
In practice
  • Implement programmatic prerequisites that block downstream tool calls until prior steps complete — e.g., block process_refund until get_customer has returned a verified customer ID.
  • Decompose multi-concern requests into distinct items, investigate each in parallel using shared context, then synthesize a unified resolution.
  • Compile structured handoff summaries (customer ID, root cause, refund amount, recommended action) when escalating to a human who can’t see the transcript.
Anti-patterns the exam tests
  • Relying on a prompt instruction (“always verify identity first”) for a rule that must hold every time — non-zero failure rate makes it unsafe for financial/irreversible actions.
  • Escalating with a bare “please help this customer” and no compiled context — the human starts from zero.
  • Treating a multi-concern request as one blob instead of decomposing it into separately-investigated items.
Exam lens The classic item: an agent occasionally issues a refund without verifying the customer, despite a system-prompt rule to verify first. The tempting answer strengthens the prompt wording. The correct answer is a prerequisite gate in code that makes process_refund impossible to call until get_customer has returned a verified ID — deterministic enforcement, not a better sentence.
⚑
Build it in a capstone The Customer Support Resolution Agent lab builds exactly this — the guarded process_refund with a confirmation gate and permission-evaluation order, plus a structured escalation handoff.
Check your understanding
  • Why is a prompt instruction insufficient for “verify identity before refund”?
  • What belongs in a mid-process escalation handoff, and why?
  • How do you handle a request that raises three separate issues at once?
Section 1.5

Apply Agent SDK hooks for tool-call interception and data normalization

Task Statement 1.5

Hooks are the SDK’s interception points — code that runs at a lifecycle moment to transform a tool result before the model sees it, or to block an outgoing tool call before it runs. They’re how you get deterministic guarantees where prompts only offer probabilities.

What & why

A PostToolUse hook intercepts a tool’s result and can transform it before the model processes it — ideal for normalizing heterogeneous data from different MCP tools. A tool-call-interception hook catches an outgoing call and can block it — ideal for enforcing a compliance rule like “no refund above $500.” The throughline with 1.4: hooks give deterministic guarantees, where prompt instructions give probabilistic compliance. When a business rule must hold every time, it belongs in a hook.

Knowledge points
  • PostToolUse hooks intercept tool results for transformation before the model processes them.
  • Interception hooks on outgoing tool calls enforce compliance (e.g., blocking refunds above a threshold).
  • The distinction between using hooks for deterministic guarantees versus prompts for probabilistic compliance.
In practice
  • Use PostToolUse to normalize heterogeneous formats — Unix timestamps, ISO 8601, numeric status codes from different MCP tools — into one shape before the agent reasons over them.
  • Use interception hooks to block policy-violating actions (a refund exceeding $500) and redirect to an alternative workflow (human escalation).
  • Choose hooks over prompt-based enforcement whenever a rule requires guaranteed compliance.
Anti-patterns the exam tests
  • Putting a hard compliance limit in the prompt (“never refund more than $500”) instead of a hook — same non-zero-failure problem as 1.4.
  • Making the model normalize inconsistent data formats in-context turn after turn, instead of normalizing once in a PostToolUse hook.
  • Treating hooks and prompt instructions as interchangeable — they differ precisely on the guarantee.
Exam lens Two flavors show up. One: heterogeneous timestamps confuse the agent — the answer is a PostToolUse normalization hook, not a prompt asking the model to be careful. Two: refunds occasionally exceed the policy cap — the answer is an interception hook that blocks and redirects, not a firmer instruction. If a rule must be guaranteed, the hook is the answer.
⚑
Build it in a capstone The Customer Support Resolution Agent and Developer Productivity labs both use hooks for guaranteed rules — the refund cap and the PreToolUse guards on writes and Bash commands.
Check your understanding
  • Which hook transforms a tool result, and which blocks a tool call?
  • Why normalize data formats in a hook rather than in the prompt?
  • State the one-line rule for choosing a hook over a prompt instruction.
Section 1.6

Design task decomposition strategies for complex workflows

Task Statement 1.6

Not every complex task is decomposed the same way. Some split into a fixed sequence known up front; others have to grow their plan as they discover what’s there. Choosing the right decomposition strategy for the workflow is the skill this section builds.

What & why

Two decomposition patterns, two situations. Fixed sequential pipelines (prompt chaining) suit predictable, multi-aspect work where you know the steps in advance — analyze each file, then run a cross-file pass. Dynamic adaptive decomposition suits open-ended investigation, where each step’s findings generate the next subtasks — you can’t write the plan up front because you don’t yet know what you’ll find. Matching the pattern to the task is the judgment; forcing an open-ended investigation into a fixed pipeline (or over-engineering a predictable review into an adaptive one) is the error.

Knowledge points
  • When to use fixed sequential pipelines (prompt chaining) vs dynamic adaptive decomposition based on intermediate findings.
  • Prompt-chaining patterns that break a review into sequential steps (analyze each file individually, then a cross-file integration pass).
  • The value of adaptive investigation plans that generate subtasks from what’s discovered at each step.
In practice
  • Select prompt chaining for predictable multi-aspect reviews; dynamic decomposition for open-ended investigation.
  • Split large code reviews into per-file local analysis passes plus a separate cross-file integration pass to avoid attention dilution.
  • Decompose an open-ended task (“add comprehensive tests to a legacy codebase”) by mapping structure first, identifying high-impact areas, then building a prioritized plan that adapts as dependencies surface.
Anti-patterns the exam tests
  • Forcing an open-ended investigation into a rigid fixed pipeline — the plan can’t adapt to what’s found.
  • Reviewing a large multi-file change in one undivided pass — attention dilutes and cross-file issues get missed or findings contradict.
  • Building an adaptive, agentic plan for a simple predictable task that a straight chain would handle — needless complexity.
Exam lens A scenario gives a task and asks how to decompose it. The tell is predictability: a known set of aspects → prompt chaining; an open-ended “explore and figure out what matters” → adaptive decomposition. For big reviews, the expected answer is per-file passes plus a cross-file integration pass — the “one giant pass” option is the distractor.
⚑
Build it in a capstone The Developer Productivity lab decomposes “understand this legacy system” adaptively (map, then investigate high-impact areas), while the Code Review and CI/CD labs use per-file-plus-integration chaining.
Check your understanding
  • What property of a task pushes you toward adaptive rather than fixed decomposition?
  • Why split a large review into per-file plus cross-file passes?
  • How would you decompose “add tests to a legacy codebase”?
Section 1.7

Manage session state, resumption, and forking

Task Statement 1.7

Long-running work spans sessions, and branches into alternatives. Knowing how to resume a prior session, when to fork one, and — the subtle part — when not to resume because the prior context has gone stale, is what keeps extended work reliable.

What & why

Named session resumption (--resume) continues a specific prior conversation. fork_session creates an independent branch from a shared analysis baseline, so you can explore divergent approaches without them interfering. The judgment the exam probes: resumption is right when prior context is mostly valid, but when prior tool results have gone stale — files changed since — starting fresh with a structured summary is more reliable than resuming with outdated results. And if you do resume after code changed, you must tell the agent what changed so it re-analyzes the right files rather than trusting stale reads.

Knowledge points
  • Named resumption with --resume to continue a specific prior conversation.
  • fork_session for independent branches from a shared baseline to explore divergent approaches.
  • The importance of informing the agent about changes to previously analyzed files when resuming after modifications.
  • Why starting fresh with a structured summary can beat resuming with stale tool results.
In practice
  • Use --resume with session names to continue named investigations across work sessions.
  • Use fork_session to create parallel exploration branches (comparing two testing strategies or refactoring approaches from a shared codebase analysis).
  • Choose between resumption and fresh-with-summary: resume when prior context is mostly valid; start fresh with an injected summary when prior tool results are stale.
  • When resuming after edits, tell the agent which files changed for targeted re-analysis rather than full re-exploration.
Anti-patterns the exam tests
  • Resuming a session whose tool results are stale (files changed) and trusting them — the agent reasons over outdated reads.
  • Resuming after edits without telling the agent what changed — it won’t know to re-analyze.
  • Using one linear session to explore two divergent approaches instead of fork_session — the branches contaminate each other.
Exam lens A scenario resumes a days-old session after the code was refactored, and the agent gives answers based on the old structure. The tempting answer is “just --resume.” The better answer recognizes the tool results are stale: either start fresh with a structured summary, or resume and explicitly inform the agent which files changed. For “compare two approaches,” fork_session from a shared baseline is the expected call.
⚑
Build it in a capstone The Developer Productivity lab’s legacy-exploration work is where resumption and stale-context judgment matter most — long sessions across a large codebase that changes underneath you.
Check your understanding
  • When is starting fresh with a summary better than --resume?
  • What must you tell a resumed agent after files have changed?
  • What does fork_session give you that a linear session can’t?
Module 1 · Wrap-up

What this domain comes down to

Agentic Architecture & Orchestration is the exam’s largest domain because it’s the foundation: the agentic loop driven by stop_reason (1.1), coordinated into hub-and-spoke multi-agent systems (1.2) with explicit context passing through the Task tool (1.3), made reliable with programmatic enforcement and hooks rather than hopeful prompts (1.4–1.5), decomposed by the pattern that fits the task (1.6), and sustained across sessions with sound resumption and forking judgment (1.7).

One thread ties most of the anti-patterns together: deterministic mechanisms beat probabilistic ones for anything that must hold. Drive loops off stop_reason, not parsed text. Enforce prerequisites in gates, not prompt sentences. Guarantee compliance with hooks, not instructions. Pass context explicitly, because nothing is inherited. If you internalize that one idea, most of this domain’s scenario questions answer themselves.
