Before you start

You should be comfortable in a terminal and reading code — this lab assumes the developer experience the exam itself assumes (roughly 6+ months building with Claude). It does not stop to explain async functions or what a system prompt is.

  • Python 3.10+ or Node 18+ — the Agent SDK ships for both. Examples lead in Python; the shape is identical in TypeScript.
  • An Anthropic API key exported as ANTHROPIC_API_KEY, with web search enabled in your Console (an admin toggle). Bedrock/Vertex auth also works.
  • A couple of local documents (a PDF or two, some markdown) for the analyzer subagent to read — so you can see document analysis alongside web research.
Verify before you build The Agent SDK, its subagent API, and the built-in web-search / web-fetch tools have all moved recently and continue to. Package names, the agents parameter shape, tool version strings (web_search_20250305 and newer), and citation fields carry real shelf-life. Wherever you see a Verify note, check it against the current docs at code.claude.com/docs and platform.claude.com/docs. The shape of each pattern is stable; exact identifiers drift.
Orientation

What you’re building, before you build it

By the end of this lab you’ll have one working program: a coordinator agent that takes a research topic (“the current state of solid-state batteries”), delegates to four specialist subagents in turn — search the web, analyze documents, synthesize findings, write the report — and produces one comprehensive report where every claim traces to a source a subagent actually returned. Each stage adds one part of that system. You are not building seven disconnected exercises; you are building one research system in seven passes.

The one thing you’re building

A multi-agent research system. A coordinator orchestrates four specialists — searcher (web), analyzer (documents), synthesizer (findings), reporter (the write-up) — each in its own context, and assembles their outputs into a cited report on any topic.

input → a research topic output → a comprehensive report guarantee → every claim is cited

How the seven stages connect

Read the diagram as a delegation flow. The coordinator (Stage 1) sits at the center and never does the research itself — it delegates. It hands the topic to the searcher, then the analyzer, then the synthesizer, then the reporter (Stages 2–4), each returning a summary rather than its raw work. Context management (Stage 5) is why that summary-only flow matters; the citation check (Stages 6–7) gates the final report so nothing uncited ships.

Don’t worry if a box uses a word you don’t know yet — the glossary below defines every term in the diagram, and each stage explains its own box in full when you reach it.

The terms this lab uses

These come up across multiple stages. Skim them now; you don’t need to memorize them — they’ll make sense as you build.

Coordinator
The top-level agent that receives the topic and delegates to specialists rather than doing the research itself. Its context stays focused on orchestration — deciding who does what and assembling the results. Sometimes called the lead or supervisor.
Subagent
A specialized child agent the coordinator spawns for a scoped task. It runs in its own fresh context, with its own system prompt, tools, and model, and returns only its final message to the coordinator. Your four: searcher, analyzer, synthesizer, reporter.
Delegation
The coordinator handing a task to a subagent. It rests on one idea: the parent receives the conclusion, not the work that produced it. The subagent’s prompt string is the only channel from parent to child.
Claude Agent SDK
The library that runs the loop, subagents, tools, and context management for you, from Python or TypeScript. The renamed successor to the “Claude Code SDK” — old names claude-code-sdk / ClaudeCodeOptions are stale.
Context isolation
The core reason subagents exist here: a searcher can read twenty pages in its own context and hand back a three-line summary, so none of that raw content lands in the coordinator’s window. Keeps the coordinator’s reasoning room free for orchestration.
WebSearch (built-in tool)
The Agent SDK’s built-in web search tool. Claude decides when to search; the API runs it and returns results with citations always attached — each cited passage carries a URL, title, and the exact cited text.
web_fetch (built-in tool)
Retrieves the full content of a specific URL or PDF. Paired with WebSearch in a “search then fetch” flow. Its citations are optional — you enable them explicitly.
Citation
A claim’s link back to the source it came from — URL, title, and the passage that supports it. In this system, a report claim without one is a defect, not a stylistic choice.
Over-delegation / subagent sprawl
The named failure modes of multi-agent systems: delegating a task too small to be worth the round-trip, or spawning so many subagents that merging their summaries costs more than the work saved. Keep the count deliberate.
One level deep
A hard boundary in the current model: a subagent cannot spawn its own subagents. The coordinator delegates; the specialists do the work. Don’t design for recursive delegation.
Synthesis
Combining findings from search and document analysis into a coherent set of claims — reconciling agreement, noting conflict — before anything is written up. The synthesizer’s job, and a data dependency: it can’t start until search and analysis finish.
Verification pass
A step that checks the finished report against the sources the subagents returned, flagging any claim that isn’t backed by one. The difference between “asked for citations” and “guaranteed them.”
Before Stage 1
  • You can state, in one sentence, what the finished system does end to end.
  • You can trace the topic through the diagram: coordinator → search → analyze → synthesize → report → citation check.
  • You can explain why the coordinator receives summaries, not raw research.
Stage 01

Stand up the coordinator

Agentic Architecture & Orchestration · 27%

The coordinator is the agent at the center of the system. Its defining trait: it does not do the research — it delegates and assembles. Build the loop first, with a system prompt that makes delegation its job, and confirm it can spawn a subagent at all.

A topic comes in: “the current state of solid-state batteries.” The coordinator needs to route it through search, analysis, synthesis, and writing — without trying to do all four itself and drowning its own context.

1a · Install and set up the coordinator

The Claude Agent SDK is the renamed successor to the old “Claude Code SDK.” If a tutorial imports claude-code-sdk or ClaudeCodeOptions, it’s stale.

Install
# Python 3.10+ pip install claude-agent-sdk # TypeScript (bundles the Claude Code binary — nothing else to install) npm install @anthropic-ai/claude-agent-sdk
The coordinator, with a delegation-first system prompt (Python)
import asyncio from claude_agent_sdk import query, ClaudeAgentOptions COORDINATOR = """You are a research coordinator. You do NOT research directly. Delegate to your specialist subagents in order: use the searcher for web sources, the analyzer for local documents, the synthesizer to combine findings, and the reporter to write the final cited report. Pass each subagent exactly what it needs. Assemble their summaries; never redo their work.""" async def main(): async for message in query( prompt="Research: the current state of solid-state batteries.", options=ClaudeAgentOptions( system_prompt=COORDINATOR, allowed_tools=["Agent"], # REQUIRED or delegation is blocked max_turns=20, # orchestration takes several turns ), ): if hasattr(message, "result"): print(message.result) asyncio.run(main())

1b · The one rule that trips everyone up

Agent must be in allowed_tools. Without it, the coordinator’s attempts to delegate fall through to your permission callback or get denied — and Claude, unable to delegate, just does the work itself in one context. If your “multi-agent” system is quietly running as a single agent, this is almost always why. You can also nudge delegation by naming a subagent in the prompt (“Use the searcher to…”).

1c · Cap the orchestration

The coordinator carries the full conversation plus every subagent’s description on every turn, so long runs get progressively more expensive. Set max_turns as a circuit breaker and watch cost — every ResultMessage carries token usage with a per-model breakdown when subagents run.

Exam lens Know the supervisor/coordinator pattern: one lead decomposes and delegates, specialists work on non-overlapping subtasks, the lead aggregates. Know that Agent in allowedTools is what makes delegation actually happen, and that the coordinator’s value is keeping its context on orchestration. “The coordinator does the research and also coordinates” is the wrong shape.
Verify The agents/Agent-tool mechanics and ClaudeAgentOptions fields are current but the SDK is pre-1.0 and moving. Confirm the import path, the allowed_tools=["Agent"] requirement, and delegation setup against code.claude.com/docs/en/agent-sdk/subagents.
Stage 1 done when
  • Your coordinator runs with a system prompt that makes delegation its job.
  • Agent is in allowed_tools, and you’ve confirmed it can spawn at least one subagent.
  • max_turns is set and you can read token usage from the result.
Stage 02

Define the four subagents

Agentic Architecture & Orchestration · 27%

Each specialist is a subagent with its own role, its own system prompt, and — crucially — a description the coordinator reads to decide when to use it. Define all four in the agents parameter, each scoped to exactly its job.

2a · The four roles

Define them programmatically. Each runs in a fresh context and returns only its final message to the coordinator.

options = ClaudeAgentOptions( system_prompt=COORDINATOR, allowed_tools=["Agent"], agents={ "searcher": { "description": "Searches the web for sources on a topic. Use FIRST, to gather cited web sources.", "prompt": "You find and return web sources with citations. Return a list of claims, each with its source URL and the cited passage. Do not synthesize.", "tools": ["WebSearch", "WebFetch"], }, "analyzer": { "description": "Analyzes LOCAL documents (PDFs, markdown). Use to extract claims from files, not the web.", "prompt": "You read local documents and extract relevant claims, each tagged with the file and location it came from.", "tools": ["Read", "Grep"], }, "synthesizer": { "description": "Combines search + document findings into coherent claims. Use AFTER search and analysis.", "prompt": "You merge findings into a deduplicated claim set. Note agreement and conflict. Preserve every claim's source. Do not write prose.", "tools": [], # reasons over what it's given; needs no tools }, "reporter": { "description": "Writes the final report from synthesized claims. Use LAST.", "prompt": "You write a comprehensive report. Every claim MUST carry its citation. Never state a claim you cannot cite.", "tools": ["Write"], }, }, )

2b · Descriptions are how the coordinator routes

The coordinator reads each description exactly the way a model reads a tool description — to decide when to delegate. So the descriptions must draw sharp boundaries: web vs. local documents, and ordering (searcher first, reporter last). Vague descriptions cause the coordinator to route to the wrong specialist or skip one.

2c · Scope each subagent’s tools to its role

Tool boundaries do real work here: the synthesizer has no tools (it only reasons over what it’s given), the analyzer reads local files but can’t search the web, the searcher searches but doesn’t write. A subagent that can only do its job can’t wander into another’s.

Exam lens Two tested ideas here. First, the coordinator delegates based on the subagent’s description — write it to say when to use the agent, not just what it is. Second, tool scoping per subagent is a real design lever: a specialist with a minimal tool set is safer and less likely to be mis-selected. Subagents are also one level deep — they can’t delegate further.
Verify The agents object keys (description, prompt, tools, model), the built-in tool names (WebSearch vs web_search casing), and filesystem-based .claude/agents/ definitions all vary by version. Confirm against the subagents docs before relying on the exact shape.
Stage 2 done when
  • All four subagents are defined with distinct roles and sharp, when-to-use descriptions.
  • Each subagent’s tools are scoped to exactly its job — no more.
  • The coordinator routes the topic to the searcher first, not to itself.
Stage 03

Give each subagent its tools

Tool Design & MCP Integration · 18%

The searcher and analyzer are only as good as the tools behind them. The searcher uses the SDK’s built-in web tools — which cite sources automatically — and the analyzer reads local files. Wire them up, and understand why the built-in web search is the reason citations are even possible.

3a · The searcher’s web tools cite for free

The built-in WebSearch tool is special for this system: citations are always enabled. When Claude answers from search results, each supporting passage comes back as a citation with the source url, title, and the exact cited_text. You don’t have to ask for citations — they arrive with the results. That’s the foundation the whole “every claim is cited” guarantee rests on.

# The searcher subagent's tools — WebSearch cites automatically. # A minimal standalone research agent looks like this: async for msg in query( prompt="Find recent sources on solid-state battery energy density.", options=ClaudeAgentOptions(allowed_tools=["WebSearch"]), ): ... # results come back with url + title + cited_text attached

3b · Add web_fetch for full-text (search then fetch)

Search returns snippets; when the searcher needs the full text of a promising page, web_fetch pulls it. The canonical pattern is search then fetch — search to find candidate URLs, fetch to read the best ones in depth. Unlike WebSearch, web_fetch citations are optional, so enable them:

# web_fetch citations are off by default — turn them on: { "type": "web_fetch_20250910", "name": "web_fetch", "max_uses": 5, "citations": {"enabled": True} }
Know the tool’s limits so the searcher doesn’t rely on it wrongly: web_fetch gets the initial HTML, not JavaScript-rendered content, and returns raw text (no structured extraction). For a research report over articles and PDFs that’s usually fine — but it’s a real boundary worth stating in the searcher’s prompt.

3c · The analyzer reads local documents

The analyzer’s tools are Read and Grep — it works over the PDFs and markdown you provided, extracting claims tagged with the file and location they came from. That file-and-location tag is the analyzer’s version of a citation, so document-sourced claims are as traceable as web ones.

If a subagent needs an external system instead of the built-ins — a private paper database, an internal wiki — that’s where an MCP server comes in, scoped to just that subagent. The built-in web + file tools cover this lab; MCP is the extension point when the source lives behind your own API.
Exam lens Know the built-in web tools: WebSearch cites automatically, web_fetch citations are opt-in, and the search-then-fetch pattern. Know web_fetch’s limits (no JS rendering, raw text). And know when a task needs an MCP server (a source behind your own API) versus a built-in tool — scoped per subagent.
Verify Tool version strings move fast (web_search_20250305, web_fetch_20250910, and newer dated versions with dynamic filtering), and which models support which version varies. Web search also requires a Console admin toggle. Confirm current versions and model support against platform.claude.com/docs before wiring them in.
Stage 3 done when
  • The searcher returns web claims that each carry a real url and cited_text.
  • web_fetch is enabled with citations on, and used to read at least one full page.
  • The analyzer extracts claims from your local documents, each tagged with its file and location.
Stage 04

Orchestrate the delegation

Agentic Architecture & Orchestration · 27%

Four subagents don’t make a system — the orchestration does. The coordinator has to pass each specialist what it needs, run them in the right order, and avoid the two classic failure modes: over-delegation and subagent sprawl.

The synthesizer can’t combine findings that don’t exist yet. So the coordinator must run search and analysis before synthesis, and synthesis before the report — a sequence driven by data dependency, not preference.

4a · Pass context explicitly — the prompt is the only channel

A subagent starts with a fresh context and no access to the coordinator’s conversation. The only thing that crosses from parent to child is the prompt string the coordinator sends. So whatever a subagent needs — the topic, the searcher’s findings for the synthesizer, the synthesized claims for the reporter — must be put directly in that prompt. This is the single most common orchestration bug: assuming a subagent “knows” what the coordinator knows.

4b · Sequence by data dependency

This system is deliberately sequential, because each step feeds the next:

  1. searcher → web claims with citations
  2. analyzer → document claims with file locations
  3. synthesizer → both sets merged into one coherent claim set (can’t start until 1–2 are done)
  4. reporter → the written report (can’t start until 3 is done)

The coordinator enforces this order — it doesn’t hand the reporter anything until the synthesizer has returned. Sequence isn’t a style choice; it’s the shape of the data flow.

4c · Avoid over-delegation and sprawl

Two named failure modes the exam cares about:

  • Over-delegation: handing a subagent a task so small the coordinator could have done it in fewer turns than the round-trip costs. If the coordinator has to do real work to use a subagent’s result, the delegation was wrong.
  • Subagent sprawl: spawning more agents than the work needs. Every subagent costs a full context window and round-trip overhead; too many, and merging their summaries costs more than you saved. Four specialists, each clearly earning its place, is the right size here.
Exam lens This is the central thread of the Orchestration domain. Know that the prompt string is the only parent→child channel (so context must be passed explicitly), that sequencing follows data dependency, and the two failure modes — over-delegation and sprawl — by name. Scenarios describe a multi-agent system misbehaving; the fix is usually “pass the context in the prompt” or “that task didn’t need a subagent.”
Verify How the coordinator invokes subagents in sequence, and how it threads one subagent’s output into the next one’s prompt, depends on current SDK delegation mechanics. Confirm the delegation and context-passing details against code.claude.com/docs/en/agent-sdk/subagents.
Stage 4 done when
  • Each subagent receives, in its prompt, exactly the context it needs from the prior step.
  • The four run in dependency order — synthesis after search + analysis, report after synthesis.
  • You can explain why this system uses four subagents and not fourteen.
Stage 05

Keep the coordinator’s context small

Context Management & Reliability · 15%

A research run reads a lot — dozens of pages, several documents. If all of that landed in the coordinator’s context, it would fill fast and the coordinator would lose the thread of what it’s orchestrating. The whole reason for subagents here is that they don’t let that happen.

5a · Summaries in, not raw work

This is the core idea, worth stating plainly: a subagent does the noisy work — the twenty page reads, the document crawl — entirely inside its own context, and returns one thing to the coordinator: a summary. The coordinator pays for the three-line finding, not the twenty pages behind it. That asymmetric information flow is the point of delegation, not a side benefit.

# The searcher reads 20 pages in ITS context and returns: { "claims": [ {"claim": "Solid-state cells reached ~400 Wh/kg in 2025 lab tests", "url": "https://…", "cited_text": "…achieved 400 Wh/kg…"}, ... ] } # The coordinator never sees the 20 pages — only this.

5b · Watch the orchestrator’s own growth

Even with summaries, the coordinator’s context grows: every turn carries the full conversation plus all four subagent descriptions. On a long research run this adds up. Keep subagent descriptions tight, keep the number of subagents deliberate (Stage 4), and reset the session between unrelated topics rather than letting one coordinator accumulate three reports’ worth of context.

5c · A scratchpad for the claim set

For a big report, keep the growing claim set on disk rather than only in the coordinator’s window — the synthesizer writes claims to a file, the reporter reads them back. State lives in the file; the coordinator’s context holds only what the current delegation step needs. It also means a long run survives a session reset.

Exam lens The reliable theme: delegation is context management. The subagent isolates the noisy work and returns a summary, keeping the coordinator’s window free for orchestration. Know why the coordinator gets summaries not raw output, that its context still grows with subagent descriptions, and that external state (a claims file) beats holding everything in-context. This is the same principle as fact-extraction, applied to a multi-agent shape.
Stage 5 done when
  • Each subagent returns a compact summary; the coordinator never accumulates raw pages.
  • You’ve checked token usage and can see the coordinator’s context staying lean across steps.
  • The claim set lives in a file the synthesizer writes and the reporter reads.
Stage 06

Enforce cited claims

Context Management & Reliability · 15%

“Comprehensive, cited reports” is only real if it’s enforced. Asking the reporter nicely to cite everything isn’t a guarantee — a model can still assert an unsupported claim. You need every claim to trace to a source a subagent actually returned, and a check that flags the ones that don’t.

The reporter writes: “Solid-state batteries will be in mass-market EVs by 2027.” Confident, plausible — and backed by none of the sources the searcher returned. That’s the exact failure the citation check exists to catch.

6a · Carry the source with every claim, end to end

The guarantee is built by construction, not bolted on at the end. Each claim carries its source from the moment it’s found: the searcher attaches url + cited_text, the analyzer attaches file + location, the synthesizer preserves those through the merge, and the reporter is instructed never to state a claim it can’t cite. By the time the report exists, every claim already has a source attached — or it shouldn’t be there.

6b · The verification pass

Construction isn’t enough on its own, because a model can still slip in an unsupported claim. So add a verification step: check the finished report against the set of sources the subagents returned, and flag any claim without a match. This is the difference between “asked for citations” and “guaranteed them.”

# Verification: every report claim must map to a returned source. returned_sources = collect_sources(searcher_out, analyzer_out) # urls + file refs for claim in report_claims: if claim.citation not in returned_sources: flag(claim, "UNSUPPORTED — no source backs this") # send back for a source or cut

6c · Make it enforcement, not guidance

The reliable pattern — the same one that matters across all these systems — is that a hard requirement belongs in a deterministic check, not only in a prompt. A SubagentStop-style gate or a post-generation verification function enforces “no uncited claims” regardless of what the reporter decided. Put the guarantee where it holds even when the model doesn’t cooperate.

Exam lens The through-line of the reliability domain: a prompt is guidance; a check is a guarantee. “Tell the model to always cite” is the tempting wrong answer when the requirement is hard. The right shape carries the source with each claim and verifies the output against the returned sources. Unsupported-but-confident claims (fabrication) are the named risk this guards against.
Verify Gating mechanisms (SubagentStop hooks, output validation) and how to read the structured citation fields (web_search_result_location, cited_text) out of tool results evolve with the SDK. Confirm the current hook events and citation response shape before wiring the check.
Stage 6 done when
  • Every claim carries its source from search/analysis through synthesis to the report.
  • A verification pass flags any report claim not backed by a returned source.
  • The “no uncited claims” rule lives in a check, not only in the reporter’s prompt.
Stage 07

Assemble & verify the report

Reliability in practice

The last stage brings it together: the reporter turns the synthesized, sourced claims into a comprehensive report, the verification pass confirms every claim is backed, and you run the whole system end to end on a real topic to see it produce something you’d actually trust.

7a · The reporter writes from sourced claims only

The reporter receives the synthesizer’s merged claim set — each claim already carrying its source — and writes the report around them: an overview, the findings organized by theme, the areas where sources conflict, and a references section. Its one hard rule, from Stage 2: never state a claim it can’t cite. It’s writing up sourced claims, not generating new ones.

7b · Run the full pipeline

Give the coordinator a real topic and watch the whole flow: it delegates to the searcher (web sources with citations), the analyzer (document claims), the synthesizer (one merged claim set), then the reporter (the write-up) — and finally the verification pass. A claim that fails verification goes back for a source or gets cut before the report is final.

7c · Judge it like a reader would

“Comprehensive and cited” has a testable meaning: pick any claim in the finished report and follow its citation to a source the searcher or analyzer actually returned. If every claim survives that trace, the system did its job. If one doesn’t, the verification pass should have caught it — and that gap is where you tune (usually the synthesizer dropping a source, or the reporter over-reaching past its claims).

Exam lens The exam frames reliability as verifiable: a definition of “done” you can check, not a hope. For this system, “every claim traces to a returned source” is that definition. Expect a scenario where the report looks great but a claim can’t be traced — the fix is upstream (preserve sources through synthesis), not “ask the reporter to try harder.”
Stage 7 done when
  • The system produces a comprehensive report on a real topic, organized and readable.
  • Every claim in it traces to a source a subagent actually returned.
  • A deliberately unsupported claim is caught by verification before the report is final.
Wrap-up

What you built, and how it maps

You now have one working multi-agent system that exercises the scenario’s three primary domains: a coordinator orchestrating four specialist subagents with scoped roles and sequencing (Agentic Architecture & Orchestration, 27%); built-in web tools that cite automatically plus scoped tool boundaries per subagent, with MCP as the extension point (Tool & MCP, 18%); and delegation-as-context-management with an enforced, verified citation guarantee (Context & Reliability, 15%). Completing it, you should be able to answer any question the scenario raises — why the coordinator delegates rather than researches, how context passes to a subagent, why the flow is sequential, and how “every claim is cited” becomes a guarantee rather than a hope.

The exam is 60 scenario questions in 120 minutes, and it rewards judgment over recall — the wrong answers are written to tempt people with partial knowledge. Having actually built the coordinator, felt why Agent must be in the allowlist, and enforced the citation check is the difference between recognizing the right architectural call and guessing.
