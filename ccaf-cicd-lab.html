Put Claude Code
in your pipeline.

A single hands-on capstone: Claude Code running headless in your CI/CD pipeline — reviewing pull requests, generating test cases, and posting feedback automatically. The hard part isn’t wiring the YAML; it’s designing prompts that give actionable feedback and don’t drown developers in false positives. You build the pipeline, then calibrate the reviewer until people actually trust it.

Before you start

You should be comfortable with a CI system and a terminal — this lab assumes the developer experience the exam itself assumes. It does not stop to explain what a GitHub Actions workflow file is or how secrets work.

  • Claude Code installed, and an Anthropic API key for CI use — a dedicated key stored as a repository or org secret (ANTHROPIC_API_KEY), separate from your personal dev key so you can track and rotate it.
  • A repo with real pull requests and a test suite — ideally one with recent merged PRs you can calibrate against in Stage 7.
  • Permission to add a workflow file and configure token scopes on that repo.
Verify before you build Headless flags, the first-party claude-code-action, output formats, and token-scope behavior all move fast. Flag names (-p, --output-format, --allowedTools, --max-turns), action versions, and env-scrub variables carry real shelf-life. Wherever you see a Verify note, check it against the current docs at code.claude.com/docs. The shape of each pattern is stable; exact identifiers drift.
Orientation

What you’re building, before you build it

By the end of this lab you’ll have one working pipeline: a CI/CD workflow that runs Claude Code headless on every pull request, reviews the diff, generates test cases for uncovered code, and posts feedback developers actually act on. The build has two halves that matter equally — the mechanics (running Claude non-interactively in CI, safely) and the prompting (feedback that’s actionable and low-noise). Each stage adds one piece. You are not doing seven disconnected exercises; you are building and calibrating one pipeline in seven passes.

The one thing you’re building

A CI/CD pipeline with Claude Code in it. On each PR it runs headless (claude -p), reviews the diff with a calibrated prompt, generates tests in a separate job, and posts severity-tagged, actionable comments — tuned so developers trust it rather than dismiss it.

input → a pull request output → actionable review + tests target → low false-positive rate
Two ways to do this, and they’re both valid. Anthropic offers a managed Code Review service (multi-agent, confidence-filtered, zero prompt-writing) and the option to run Claude yourself in CI via headless mode + the claude-code-action. This lab builds the self-hosted path, because that’s where the prompt-engineering and configuration judgment the exam tests actually lives. The managed service is noted where relevant.

How the seven stages connect

Read the diagram as a pipeline triggered by a PR. Claude Code runs headless (Stage 1) inside a CI job (Stage 2). What it produces is governed by the review prompt (Stage 3) and the false-positive controls (Stage 4); a separate job generates tests (Stage 5). Project context (Stage 6) feeds the prompts, and calibration (Stage 7) is the loop that tunes the whole thing against real PRs until developers trust it.

Don’t worry if a box uses a word you don’t know yet — the glossary below defines every term in the diagram, and each stage explains its own box in full when you reach it.

The terms this lab uses

These come up across multiple stages. Skim them now; you don’t need to memorize them — they’ll make sense as you build.

Headless mode (-p / --print)
Claude Code without an interactive session: pass one prompt via -p, it runs the full agent loop and exits with a result. This is what makes Claude Code runnable in any CI system. The single most common CI mistake is omitting -p — the job hangs waiting for input that never comes.
Output format
How headless results come back: text (debugging), json (programmatic, parse with jq), or stream-json (real-time). Choose json when a later step reads the result.
Exit code
Headless Claude returns 0 on success and non-zero on failure, so it composes with &&/|| in shell scripts and can gate a pipeline step.
claude-code-action
Anthropic’s first-party GitHub Action that handles auth, rate limiting, and output formatting for you — the drop-in way to run Claude in a GitHub Actions workflow.
Managed Code Review
Anthropic’s hosted review service: multiple agents analyze the diff in parallel, a verification step filters false positives, and findings post as severity-ranked inline comments — no prompt-writing. An alternative to the self-hosted path this lab builds.
Fork safety
PRs from forks don’t get your repository secrets by default — so an untrusted contributor can’t trigger your workflow and spend your API credits. Supporting fork PRs (pull_request_target) needs great care about what code you check out.
Token scope / least privilege
The GitHub token’s permissions. A workflow that only posts comments needs pull-requests: write and contents: read — nothing more. Adding a permissions block resets every unnamed scope to none.
False positive
A finding that isn’t a real problem — a flagged non-issue. The metric that decides whether developers trust the reviewer: too many, and they start ignoring all of it. Minimizing these is the central prompt-engineering task.
Confidence scoring
Having the review rate each candidate finding 1–10 and post only the high-confidence ones (Anthropic’s own tooling uses a threshold like ≥8). The primary lever for cutting noise.
Verification step
Requiring evidence before a finding posts — a file:line citation, a concrete attack path — rather than an inference from naming. Filters out plausible-but-wrong flags.
Severity
A label on each finding (blocker / major / minor / nit) so developers can triage at a glance. Consistent severity labels, plus a cap on nits, keep reviews actionable.
CLAUDE.md / REVIEW.md
Project files that give the CI-invoked reviewer the same context a human reviewer would have — conventions, what to focus on, what to skip. REVIEW.md is injected at highest priority for review rules specifically.

Stage 01

Run Claude Code headless

Claude Code Configuration & Workflows · 20%

Everything in this lab rests on one flag. Interactive Claude Code waits for you; a pipeline has no one to wait for. Headless mode runs a single prompt and exits — and getting this right (or wrong) is the difference between a working job and one that hangs forever.

Your pipeline runs claude "review this PR" and the job hangs indefinitely. The logs show it’s waiting for interactive input. You try redirecting stdin, setting an env var — nothing works. The fix is one flag: -p.

1a · The -p flag is headless mode

-p (short for --print) runs Claude Code non-interactively: it processes the prompt, runs the full agent loop (thinking, tool calls, edits), prints the result to stdout, and exits. No confirmations, no waiting. This is the way Claude Code runs in CI, pre-commit hooks, and any script.

# Pipe the PR diff in, get a review out, then exit. gh pr diff 123 | claude -p "Review this diff for bugs, security, and correctness."
This is a known exam item: a pipeline hangs because -p is missing. Invented alternatives — a CLAUDE_HEADLESS env var, an stdin redirect, a --batch flag — are not real Claude Code features. The fix is always -p.

1b · Choose an output format

Headless mode emits three shapes: text for quick debugging, json for programmatic processing (parse with jq), and stream-json for real-time feedback on long tasks. When a later pipeline step reads the result, use JSON.

claude -p "$(cat review-prompt.txt)" --output-format json --max-turns 3

1c · Exit codes compose with the shell

Headless Claude returns 0 on success and non-zero on failure, so it drops straight into shell logic with && and || — run the review, and only post if it succeeded; or fail the step if Claude errored. Pair this with --max-turns (bound the loop) and --allowedTools (scope what it can do) to keep CI runs bounded and cheap.

Exam lens Know cold that -p is what makes Claude Code work in CI, and that a hanging pipeline means it’s missing. Know the three output formats and when to use JSON, and that exit codes make it scriptable. --max-turns and --allowedTools as cost/scope bounds are the other reliable items here.
Verify -p, --output-format, --max-turns, and --allowedTools are current but the CLI moves fast. Confirm flag names and output-format values against code.claude.com/docs before wiring a pipeline.
Stage 1 done when
  • You can run a review headless with -p and get a result without the process hanging.
  • You’ve produced JSON output and parsed a field out of it.
  • --max-turns and --allowedTools bound the run.
Stage 02

Wire it into the pipeline

Claude Code Configuration & Workflows · 20%

Now put the headless call inside a real workflow that triggers on pull requests, posts back to GitHub, and does so safely. Two things bite teams here: token permissions and fork safety. Get them right and the pipeline is both functional and secure.

2a · A minimal review workflow

Trigger on PR events, check out the code, get the diff, run Claude headless, post the result. You can use the first-party anthropics/claude-code-action (it handles auth, rate limiting, and formatting) or call claude -p directly.

on: pull_request: types: [opened, synchronize] paths-ignore: ["**.md", "docs/**"] # don't review docs-only PRs permissions: contents: read # read the code pull-requests: write # post comments — nothing more jobs: review: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: | gh pr diff ${{ github.event.number }} | \ claude -p "$(cat .github/review-prompt.txt)" --output-format json > out.json gh pr comment ${{ github.event.number }} --body-file review.md env: ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}

2b · Scope the token to least privilege

The GitHub token’s permissions should be exactly what the job needs, no more. A reviewer that posts comments needs contents: read and pull-requests: write — not contents: write, not issues unless it touches them. One sharp gotcha: the moment you add a permissions block, GitHub resets every scope you don’t name to none — so a job that posts a comment with only contents: read gets rejected. Name every scope you use.

2c · Fork safety and secret hygiene

Two security facts that matter in CI:

  • Fork safety: PRs from forks don’t receive your repository secrets by default — which is what stops an untrusted contributor from triggering your workflow and burning your API credits. Supporting fork PRs means pull_request_target, and then you must be extremely careful about what code you check out and run.
  • Secret hygiene: the API key lives in secrets, never in the YAML or logs. GitHub masks secrets automatically, but in custom scripts pass the key via the environment, never on the command line where it lands in the process list. CLAUDE_CODE_SUBPROCESS_ENV_SCRUB=1 strips sensitive env vars from Claude’s subprocess context so it can’t read or leak them.
Exam lens Two heavily-tested CI ideas: least-privilege token scopes (and that a permissions block zeroes unnamed scopes — a classic “why is my comment rejected” answer), and fork safety (secrets withheld from fork PRs; pull_request_target is the risky opt-in). Path filters to skip docs-only PRs are a standard cost control.
Verify The claude-code-action version and inputs, token-scope reset behavior, and CLAUDE_CODE_SUBPROCESS_ENV_SCRUB are current but evolve. Confirm the action inputs and the GitHub Actions permission model against the current docs.
Stage 2 done when
  • A workflow triggers on PRs, runs Claude headless, and posts a comment back.
  • Token scopes are minimal, and you named every scope the job uses.
  • You can explain why fork PRs can’t spend your credits by default.
Stage 03

Write the review prompt

Prompt Engineering & Structured Output · 20%

With Claude running in the pipeline, the prompt becomes the review strategy. In CI there’s no back-and-forth to clarify — the prompt must contain everything. A lazy prompt gives a noisy, generic dump; a sharp one gives a review that reads like a senior engineer who actually read the code.

“Review this code” produces a laundry list of style nitpicks nobody asked for. The same diff with a focused, structured prompt produces three real bugs with line numbers and fixes. Same model — the difference is entirely the prompt.

3a · The three levers: role, focus, format

Every good review prompt sets three things:

  • Role — who Claude is reviewing as (“a senior engineer applying our standards”), which sets the bar.
  • Focus — what to inspect and what to ignore (“correctness, security, and edge cases; not style — the linter owns that”). Naming the failure modes to hunt beats asking for “feedback.”
  • Format — the exact output shape: severity, rationale, a file:line, and a suggested fix — ready to post as a GitHub comment, not an abstract opinion.

3b · A review prompt that produces postable output

You are reviewing a pull request as a senior engineer applying our conventions. Inspect ONLY for: correctness and logic errors, security issues, broken edge cases, and missing error handling. Do NOT comment on style or formatting — the linter handles those. For each issue, output: - severity: blocker | major | minor | nit - location: file:line - problem: one sentence - fix: a concrete suggestion Separate CONFIRMED issues from POSSIBLE risks. If you are not certain an issue is real, put it under POSSIBLE and say what you're unsure about. If there are no confirmed issues, say so plainly.

3c · Ask it to express uncertainty, not hide it

The single easiest noise-reducer: tell Claude to separate confirmed issues from possible risks, and to never present a guess as a fact. A reviewer that says “I’m not certain, but check whether this handles null” is useful; one that states a confident falsehood is worse than silence. This also sets up the confidence machinery in Stage 4.

Exam lens Know the role / focus / format structure, and that in CI the prompt must be self-contained (no clarifying turns). Asking for severity-ranked, evidence-tied, ready-to-post findings — and separating confirmed from possible — is the canonical “actionable feedback” pattern. “Just ask it to review the code” is the distractor.
Stage 3 done when
  • Your prompt sets role, focus (with explicit exclusions), and a postable output shape.
  • It separates confirmed issues from possible risks and forbids presenting guesses as facts.
  • Run on a real diff, it produces severity-tagged findings with line numbers and fixes.
Stage 04

Minimize false positives

Prompt Engineering & Structured Output · 20%

This is the stage that decides whether the pipeline survives. A reviewer that cries wolf gets muted — once developers dismiss a few bad flags, they stop reading all of them, and the whole investment is wasted. Minimizing false positives isn’t polish; it’s what makes the reviewer trusted.

The reviewer flags a “SQL injection” that’s actually a parameterized query, and a “race condition” that’s theoretical. Two bad flags on one PR, and the author starts ignoring every comment the bot posts. That’s the failure this stage prevents.

4a · Confidence scoring with a threshold

The primary lever, straight from Anthropic’s own review tooling: have the review assign each candidate finding a confidence score (say 1–10) and post only the high-confidence ones. The security-review tooling filters anything below 8. Low-confidence candidates become noise; drop them before they reach a developer.

For each candidate finding, assign a confidence score 1–10: 1–3 likely false positive or noise → DISCARD 4–7 uncertain → list under POSSIBLE, do not post inline 8–10 high confidence → post as a finding Ask yourself: would this finding be actionable for the team? If not, discard it.

4b · Require verification, not inference

The second lever: demand evidence before a finding posts. A behavior claim needs a file:line citation in the actual source, not an inference from a function’s name. A security finding needs a concrete, specific attack path, not a theoretical one. This verification bar is exactly what the managed Code Review service does — a step that checks candidates against actual code behavior before posting.

4c · Explicit exclusions for your codebase

The third lever: tell the reviewer what not to flag, based on what generates noise in your code. Anthropic’s own security review prompt is full of these — don’t report memory-safety issues in memory-safe languages, don’t flag client-side auth (the server handles it), don’t report theoretical race conditions, skip test-only files. Add your codebase’s known non-issues (“don’t flag any in /src/legacy”), and cap nits (“report at most five nits; summarize the rest as a count”).

Skip rules keep reviews focused: list paths and categories where Claude posts nothing — generated code, lockfiles, vendored dependencies, and anything your CI already enforces (lint, spellcheck). No point paying Claude to re-flag what a linter already caught.
Exam lens This is the most tested idea in the scenario. The three levers — confidence threshold, verification/evidence bar, and explicit exclusions — are all ways to cut false positives. Know the “would this be actionable?” test and that a noisy reviewer gets ignored entirely. “Tell it to try harder” or “flag everything to be safe” are the wrong answers — over-flagging is the failure, not under-flagging.
Verify The confidence-threshold convention (≥8) comes from Anthropic’s published security-review tooling and is a strong default, but tune it to your codebase. The managed Code Review’s verification behavior may change; confirm against code.claude.com/docs/en/code-review.
Stage 4 done when
  • Your prompt scores findings by confidence and posts only high-confidence ones.
  • Findings require a file:line evidence citation, not an inference.
  • Explicit exclusions and a nit cap keep the output focused on what matters.
Stage 05

Generate test cases

Prompt Engineering & Structured Output · 20%

The second job the pipeline does: generate tests for code that lacks them. This is a distinct task from review — and it belongs in a separate CI step with its own fresh session, not bolted onto the reviewer.

A PR adds a new function with no test. The pipeline should notice the coverage gap, generate a test that actually exercises the function’s behavior and edge cases, and open it for the author — without contaminating the review job’s reasoning.

5a · Keep generation and review in separate sessions

Each claude -p invocation is a fresh session with no memory of the others — and in CI that isolation is the correct architecture, not a limitation. Run test generation as its own step so its context doesn’t bleed into the review (a reviewer that just wrote the tests isn’t an independent reviewer of them).

- name: Review run: gh pr diff $PR | claude -p "$(cat review-prompt.txt)" --output-format json - name: Generate tests run: claude -p "$(cat testgen-prompt.txt)" --allowedTools "Read,Write,Bash" # separate step, separate session — no shared reasoning

5b · Prompt for tests that exercise behavior

Weak test generation produces tests that assert the code does what it does — passing but worthless. Prompt for tests that cover the real cases: the happy path, the edge cases (empty, null, boundary), and the error conditions. Point it at the project’s test conventions (framework, structure, naming) so the output fits in.

5c · Detect the gap, then fill it

The useful pattern is coverage-aware: identify which changed functions lack tests, and generate only for those. That keeps the job focused and cheap, and it mirrors how a good human reviewer thinks — “this new function needs a test” — rather than regenerating a whole suite.

Exam lens Know that each claude -p is a fresh, isolated session, and that running generation and review as separate steps is the correct architecture (independence, clean context). Know that good test-generation prompts target behavior and edge cases, not tautological pass-through tests.
Verify --allowedTools values and how the action exposes the diff/coverage to a step vary by version. Confirm the current tool names and step wiring against the docs.
Stage 5 done when
  • Test generation runs as its own CI step with a separate session from the review.
  • Generated tests exercise real behavior and edge cases, matching your test conventions.
  • The job targets the coverage gap rather than regenerating everything.
Stage 06

Give it project context

Claude Code Configuration & Workflows · 20%

A CI-invoked reviewer starts knowing nothing about your project — so it reviews generically, flagging things your team decided long ago and missing what you actually care about. CLAUDE.md (and REVIEW.md) is how you give it the context a human reviewer would already have.

6a · CLAUDE.md is your CI review config

Committed to the repo and read on every run, CLAUDE.md carries your conventions, build/test commands, and architectural rules into the pipeline. With it, the reviewer focuses on what your team cares about and skips what your existing tools already handle. Without it, every review is generic. This is the same file the rest of Claude Code uses — here it doubles as review configuration.

6b · REVIEW.md for review-specific rules, at highest priority

For rules that must land reliably on every PR — “new API routes require an integration test,” “in scripts/, only flag near-certain severe issues” — a REVIEW.md is injected at highest priority, so these are followed more reliably than the same rules buried in a long CLAUDE.md. Put the review’s must-follow rules and skip rules here.

6c · Control cost with context and caching

Context feeds cost, so manage both together: path filters skip PRs that don’t need review (docs-only); prompt caching caches the CLAUDE.md and system context that’s identical run-to-run, cutting input token cost substantially; and --max-turns keeps any single run bounded. A tighter context is both cheaper and more focused.

Exam lens Know that CLAUDE.md gives the CI reviewer human-equivalent context (without it, generic reviews), and that REVIEW.md is highest-priority for review rules. Know the cost levers — path filters, prompt caching, --max-turns — and that skip rules should offload to existing tools (linters) rather than re-checking them.
Verify REVIEW.md priority behavior, prompt-caching setup in the action, and path-filter syntax are current but evolve. Confirm against code.claude.com/docs/en/code-review and the memory docs.
Stage 6 done when
  • A committed CLAUDE.md gives the reviewer your conventions and it visibly follows them.
  • A REVIEW.md carries the must-follow and skip rules for reviews.
  • Path filters and caching are in place, and you can explain how each controls cost.
Stage 07

Calibrate on real PRs

Reliability in practice

A review pipeline isn’t “done” when it runs — it’s done when developers trust it. That’s a measurable state, and reaching it means treating the first period as calibration: run the reviewer on real PRs, watch what it flags, and tune until the signal-to-noise is right.

7a · Measure the dismissal rate

The honest metric of a review bot is how often its comments get dismissed. A useful benchmark from teams running this in production: if developers are dismissing more than ~60% of automated comments, the prompt needs work. Watch the 👍/👎 reactions (the managed service attaches them and tunes on the aggregate) or track how many comments get resolved by a real fix versus ignored.

7b · Calibrate against merged PRs

Don’t tune in the abstract. Run the reviewer on your last 20 merged PRs and compare its findings to what your human reviewers actually cared about. Where it flagged noise, add an exclusion or raise the confidence bar (Stage 4). Where it missed a real issue, sharpen the focus (Stage 3). This is the loop the diagram’s calibration arrow represents.

7c · Frame it as a first pass, not a gatekeeper

Set the expectation explicitly, in the reviewer’s own comment header: this is an automated first pass for the mechanical issues, and human review is still required for architecture, business logic, and judgment. The managed service reinforces this by design — its findings don’t approve or block the PR, so existing review workflows stay intact. An honest frame is what keeps the tool trusted instead of resented.

Exam lens The reliability framing: a CI reviewer’s success is measured (dismissal rate, resolved-vs-ignored), not assumed, and reached through calibration on real PRs. Know the “first pass, not a gatekeeper” framing and that automated findings shouldn’t hard-block merges. The tempting wrong answer treats setup as completion — the work is the tuning.
Stage 7 done when
  • You’re tracking a dismissal/resolution signal and know your current rate.
  • You’ve calibrated against recent merged PRs and tuned the prompt/controls from what you saw.
  • The reviewer presents itself as a first pass, and its findings don’t hard-block merges.
Wrap-up

What you built, and how it maps

You now have one working CI/CD pipeline that exercises the scenario’s two primary domains: Claude Code Configuration & Workflows (20%) — headless mode, the workflow, token scopes and fork safety, project context, and cost control — and Prompt Engineering & Structured Output (20%) — role/focus/format review prompts, the false-positive controls (confidence, verification, exclusions), and behavior-focused test generation. Completing it, you should be able to answer any question the scenario raises — why a pipeline hangs without -p, how to scope a token, which levers cut false positives, and why calibration is the real work.

The exam is 60 scenario questions in 120 minutes, and it rewards judgment over recall — the wrong answers are written to tempt people with partial knowledge. Having actually run Claude headless, scoped the token, and tuned a prompt down from noisy to trusted is the difference between recognizing the right call and guessing.
