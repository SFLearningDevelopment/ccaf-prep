Section 3.1

Configure CLAUDE.md files with hierarchy, scoping, and modular organization

Task Statement 3.1

CLAUDE.md is how you tell Claude Code about your project once, instead of every session. The catch is where you put it: the same instructions in the wrong file reach only you, or reach every project you touch. Hierarchy is the whole game here.

What & why

There are three levels. User-level (~/.claude/CLAUDE.md) applies to you across all your projects — and critically, it is not shared with teammates via version control. Project-level (.claude/CLAUDE.md or a root CLAUDE.md) is committed, so the whole team gets it. Directory-level CLAUDE.md files apply to work in that subdirectory. As the file grows, you keep it modular with the @import syntax to reference external standards files, or split it into topic-specific files under .claude/rules/.

Knowledge points
  • The hierarchy: user-level (~/.claude/CLAUDE.md), project-level (.claude/CLAUDE.md or root CLAUDE.md), directory-level (subdirectory files).
  • User-level settings apply only to that user — instructions in ~/.claude/CLAUDE.md are not shared with teammates via version control.
  • The @import syntax for referencing external files to keep CLAUDE.md modular.
  • .claude/rules/ as an alternative to a monolithic CLAUDE.md, organizing topic-specific rule files.
In practice
  • Diagnose hierarchy issues — e.g. a new team member not getting instructions because they live in user-level rather than project-level config.
  • Use @import to selectively include relevant standards files in each package's CLAUDE.md.
  • Split large CLAUDE.md files into focused files in .claude/rules/ (testing.md, api-conventions.md, deployment.md).
  • Use the /memory command to verify which memory files are loaded and diagnose inconsistent behavior across sessions.
Anti-patterns the exam tests
  • Putting team conventions in ~/.claude/CLAUDE.md and wondering why teammates don't follow them — user-level isn't shared.
  • Letting CLAUDE.md grow into a monolithic wall of instructions instead of using @import or .claude/rules/.
  • Guessing at why behavior differs between sessions instead of running /memory to see what's actually loaded.
Exam lens The signature scenario: a developer's conventions work for them but a new team member doesn't get them. The answer is always the hierarchy — the instructions are in user-level config and need to move to project-level so version control shares them. If a question mentions inconsistent behavior across sessions, /memory is how you diagnose it.

Section 3.2

Create and configure custom slash commands and skills

Task Statement 3.2
🔒
This section is locked.

Complete the knowledge check in Section 3.1 to unlock it.

Go to Section 3.1

Slash commands and skills are how you package a repeatable workflow so it's one invocation instead of a paragraph of instructions. The design decisions: who gets it (project vs personal), what tools it may touch, and whether its output should pollute your main conversation.

What & why

Slash commands live in .claude/commands/ (project-scoped, shared via version control) or ~/.claude/commands/ (personal). Skills live in .claude/skills/ with a SKILL.md that supports frontmatter: context: fork, allowed-tools, and argument-hint. The one to understand deeply is context: fork — it runs the skill in an isolated sub-agent context so its output doesn't pollute the main conversation, which matters enormously for verbose work. And the choice between skills and CLAUDE.md is about when the content is needed: skills are on-demand for task-specific workflows; CLAUDE.md is always-loaded universal standards.

Knowledge points
  • Project-scoped commands in .claude/commands/ (shared via version control) vs user-scoped in ~/.claude/commands/ (personal).
  • Skills in .claude/skills/ with SKILL.md frontmatter: context: fork, allowed-tools, argument-hint.
  • context: fork runs a skill in an isolated sub-agent context, preventing skill output from polluting the main conversation.
  • Personal skill customization: create variants in ~/.claude/skills/ with different names to avoid affecting teammates.
In practice
  • Create project-scoped slash commands in .claude/commands/ for team-wide availability.
  • Use context: fork to isolate verbose skills — codebase analysis, or exploratory brainstorming — from the main session.
  • Configure allowed-tools in frontmatter to restrict tool access during skill execution (e.g. limiting to file writes to prevent destructive actions).
  • Use argument-hint to prompt developers for required parameters when they invoke a skill without arguments.
  • Choose skills vs CLAUDE.md: on-demand task workflows vs always-loaded universal standards.
Anti-patterns the exam tests
  • Putting universal standards in a skill (never loaded unless invoked) or a one-off workflow in CLAUDE.md (loaded always, wasting context).
  • Running a verbose analysis skill without context: fork — its output floods the main conversation.
  • Editing a shared team skill to suit yourself instead of making a personal variant in ~/.claude/skills/.
  • Leaving allowed-tools unset on a skill that shouldn't be able to run destructive commands.
Exam lens Two reliable items. One: a skill produces pages of output that derails the session — the answer is context: fork. Two: skills-vs-CLAUDE.md — ask “is this always relevant, or only for a specific task?” Always → CLAUDE.md. Task-specific → skill. Also know that customizing a shared skill for yourself means a personal variant with a different name, not editing the team's copy.

Section 3.3

Apply path-specific rules for conditional convention loading

Task Statement 3.3
🔒
This section is locked.

Complete the knowledge check in Section 3.2 to unlock it.

Go to Section 3.2

Not every convention applies everywhere. Terraform rules matter in terraform/; test conventions matter for test files wherever they live. Path-specific rules load only when you're editing matching files — less irrelevant context, fewer tokens, sharper behavior.

What & why

Files in .claude/rules/ can carry YAML frontmatter with a paths field containing glob patterns. The rule activates only when the file being edited matches. That does two things: it cuts irrelevant context and token usage, and — the part the exam cares about — it handles conventions that span multiple directories, which a directory-level CLAUDE.md fundamentally can't. Test files scattered across a codebase are the canonical case: **/*.test.tsx catches them all regardless of location.

Knowledge points
  • .claude/rules/ files with YAML frontmatter paths fields containing glob patterns for conditional activation.
  • Path-scoped rules load only when editing matching files, reducing irrelevant context and token usage.
  • The advantage of glob-pattern rules over directory-level CLAUDE.md for conventions spanning multiple directories (e.g. test files spread throughout).
In practice
  • Create rules with path scoping — e.g. paths: ["terraform/**/*"] — so they load only for matching files.
  • Use glob patterns to apply conventions by file type regardless of directory (**/*.test.tsx for all test files).
  • Choose path-specific rules over subdirectory CLAUDE.md when conventions must apply to files spread across the codebase.
Anti-patterns the exam tests
  • Using directory-level CLAUDE.md for cross-cutting conventions — you'd have to duplicate it into every directory that contains a test file.
  • Putting narrow, path-specific conventions in the root CLAUDE.md — loaded always, relevant rarely, wasting context on every session.
Exam lens The tell is “spread across the codebase.” Conventions confined to one directory → a directory CLAUDE.md is fine. Conventions that follow a file type wherever it lives (tests, migrations, .tf files) → path-specific rules with glob frontmatter. The distractor is duplicating a directory-level file everywhere.

Section 3.4

Determine when to use plan mode vs direct execution

Task Statement 3.4
🔒
This section is locked.

Complete the knowledge check in Section 3.3 to unlock it.

Go to Section 3.3

Plan mode explores and designs before touching anything; direct execution just does the work. Using plan mode for a one-line fix is friction; skipping it on a 45-file migration is how you get expensive rework. Knowing which is which is the skill.

What & why

Plan mode is for complex tasks: large-scale changes, multiple valid approaches, architectural decisions, multi-file modifications. It enables safe codebase exploration and design before committing to changes, which prevents costly rework. Direct execution suits simple, well-scoped changes — a single validation check, a one-file bug fix with a clear stack trace. They also combine well: plan the investigation, then execute the planned approach directly. Separately, the Explore subagent isolates verbose discovery output and returns a summary, preserving your main conversation's context.

Knowledge points
  • Plan mode is designed for large-scale changes, multiple valid approaches, architectural decisions, and multi-file modifications.
  • Direct execution is appropriate for simple, well-scoped changes (adding a single validation check to one function).
  • Plan mode enables safe exploration and design before committing, preventing costly rework.
  • The Explore subagent isolates verbose discovery output and returns summaries to preserve main-conversation context.
In practice
  • Select plan mode for architectural implications — microservice restructuring, a library migration affecting 45+ files, choosing between integration approaches with different infrastructure requirements.
  • Select direct execution for well-understood changes — a single-file bug fix with a clear stack trace, adding a date validation conditional.
  • Use the Explore subagent for verbose discovery to prevent context exhaustion during multi-phase tasks.
  • Combine both: plan mode for investigation, direct execution for implementation.
Anti-patterns the exam tests
  • Plan mode for a trivial, well-scoped fix — needless ceremony on a one-line change.
  • Direct execution on a 45-file migration with several viable approaches — you commit before you understand, and pay in rework.
  • Running verbose discovery in the main session instead of delegating to the Explore subagent, exhausting context before the real work starts.
Exam lens Scenarios give you a task and ask which mode. The signals: multiple valid approaches, architectural stakes, many files → plan mode. Clear scope, one file, an obvious fix → direct execution. And when a question mentions verbose exploration eating context, the answer is the Explore subagent — isolate the noise, return a summary.

Section 3.5

Apply iterative refinement techniques for progressive improvement

Task Statement 3.5
🔒
This section is locked.

Complete the knowledge check in Section 3.4 to unlock it.

Go to Section 3.4

Prose is a lossy way to describe a transformation. Examples aren't. This section is a set of techniques for closing the gap between what you meant and what you got — and for knowing when to fix things all at once versus one at a time.

What & why

When a prose description gets interpreted inconsistently, concrete input/output examples are the most effective fix — they communicate the expected transformation exactly. Test-driven iteration writes the test suite first, then iterates by sharing failures. The interview pattern has Claude ask you questions to surface considerations you hadn't thought of, before implementing. And there's a judgment call on batching fixes: interacting problems go in a single message (fixing one affects another), while independent problems iterate sequentially.

Knowledge points
  • Concrete input/output examples as the most effective way to communicate expected transformations when prose is interpreted inconsistently.
  • Test-driven iteration: write test suites first, then iterate by sharing test failures.
  • The interview pattern: having Claude ask questions to surface considerations you may not have anticipated before implementing.
  • When to provide all issues in a single message (interacting problems) versus fixing them sequentially (independent problems).
In practice
  • Provide 2–3 concrete input/output examples when natural-language descriptions produce inconsistent results.
  • Write test suites covering expected behavior, edge cases, and performance before implementation, then iterate by sharing failures.
  • Use the interview pattern to surface design considerations (cache invalidation strategy, failure modes) before implementing in an unfamiliar domain.
  • Provide specific test cases with example input and expected output to fix edge-case handling (null values in a migration script).
  • Batch interacting fixes in one detailed message; iterate sequentially on independent ones.
Anti-patterns the exam tests
  • Re-describing a transformation in more prose when it keeps coming out wrong — examples communicate what words can't.
  • Fixing interacting issues one at a time — each fix disturbs the others; they need to be addressed together.
  • Implementing in an unfamiliar domain without an interview, then discovering the unasked questions in production.
Exam lens “I explained it three times and it still gets it wrong” → concrete input/output examples, not a fourth explanation. “Several bugs, and fixing one breaks another” → one message with all of them, because they interact. Independent bugs → sequential. The interview pattern shows up as “unfamiliar domain, unclear requirements.”

Section 3.6

Integrate Claude Code into CI/CD pipelines

Task Statement 3.6
🔒
This section is locked.

Complete the knowledge check in Section 3.5 to unlock it.

Go to Section 3.5

Claude Code in a pipeline is Claude Code without a human to answer its questions. That single fact drives everything here: the flag that makes it non-interactive, the output format a script can parse, and the context it needs because nobody's there to supply it.

What & why

The -p (or --print) flag runs Claude Code non-interactively — without it, a pipeline job hangs waiting for input that never comes. --output-format json with --json-schema produces machine-parseable structured findings you can post as inline PR comments. CLAUDE.md is how you give CI-invoked Claude the project context a human reviewer would already have. And one subtle but heavily-tested point: session context isolation — the same session that generated code is less effective at reviewing its own changes than an independent review instance.

Knowledge points
  • The -p / --print flag for non-interactive mode in automated pipelines.
  • --output-format json and --json-schema for enforcing structured output in CI.
  • CLAUDE.md as the mechanism for providing project context (testing standards, fixture conventions, review criteria) to CI-invoked Claude Code.
  • Session context isolation: the session that generated code is less effective at reviewing its own changes than an independent instance.
In practice
  • Run Claude Code in CI with -p to prevent interactive input hangs.
  • Use --output-format json with --json-schema to produce machine-parseable findings for automated posting as inline PR comments.
  • Include prior review findings in context when re-running after new commits, instructing Claude to report only new or still-unaddressed issues — avoiding duplicate comments.
  • Provide existing test files in context so test generation doesn't duplicate scenarios already covered.
  • Document testing standards, valuable test criteria, and available fixtures in CLAUDE.md to improve test generation and reduce low-value output.
Anti-patterns the exam tests
  • Running claude without -p in a pipeline — the job hangs on interactive input. (Invented alternatives like a CLAUDE_HEADLESS env var or a --batch flag are not real.)
  • Reviewing generated code in the same session that generated it — it retains the generation reasoning and won't question its own decisions.
  • Re-running reviews without prior findings — the same issues get posted again as duplicate comments.
  • Generating tests without the existing suite in context — you get duplicates of scenarios already covered.
Exam lens Three near-certain items. A hanging pipeline → add -p (the other options are invented flags). Machine-parseable findings → --output-format json with --json-schema. Self-review is weaker than an independent instance — and the fix is a separate Claude invocation, not a “review your own work carefully” instruction or extended thinking.

Module 3 · Wrap-up

What this domain comes down to

Claude Code Configuration & Workflows is about giving Claude the right context at the right time: instructions placed at the right level of the hierarchy (3.1), packaged into commands and skills with the right scope and isolation (3.2), rules that load only when they're relevant (3.3), the judgment to plan or just execute (3.4), techniques that close the gap between intent and result (3.5), and the configuration that makes it work unattended in a pipeline (3.6).

One thread ties the anti-patterns together: scope and timing beat volume. Team conventions belong where the team gets them, not in your personal config. Narrow rules load on matching paths, not always. Verbose skills fork so they don't flood the session. A reviewer works better without the generator's context. More context in the wrong place is worse than less context in the right one.
