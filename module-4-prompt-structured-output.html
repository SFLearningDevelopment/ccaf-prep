Section 4.1

Design prompts with explicit criteria to improve precision and reduce false positives

Task Statement 4.1

“Be conservative” and “only report high-confidence findings” feel like precision instructions. They aren’t — they’re vague, and vague instructions don’t improve precision. What works is telling the model exactly which categorical conditions constitute a finding.

What & why

The difference is concrete criteria versus general exhortation. “Flag comments only when claimed behavior contradicts actual code behavior” is a rule the model can apply consistently; “check that comments are accurate” is not. General instructions like “be conservative” or “only report high-confidence findings” fail to improve precision compared to specific categorical criteria. This matters because of trust: a category with a high false-positive rate doesn’t just waste time, it undermines developer confidence in the accurate categories too — people start dismissing everything.

Knowledge points
  • The importance of explicit criteria over vague instructions — “flag comments only when claimed behavior contradicts actual code behavior” vs “check that comments are accurate.”
  • General instructions like “be conservative” or “only report high-confidence findings” fail to improve precision compared to specific categorical criteria.
  • The impact of false-positive rates on developer trust: high-FP categories undermine confidence in accurate categories.
In practice
  • Write specific review criteria defining which issues to report (bugs, security) versus skip (minor style, local patterns) — rather than relying on confidence-based filtering.
  • Temporarily disable high-false-positive categories to restore developer trust while you improve the prompts for those categories.
  • Define explicit severity criteria with concrete code examples for each level to achieve consistent classification.
Anti-patterns the exam tests
  • Adding “be conservative” or “only high-confidence findings” and expecting precision to improve — these are the named ineffective instructions.
  • Leaving a noisy category running while you work on it — it erodes trust in the categories that are accurate.
  • Severity labels without concrete criteria — classification drifts between runs.
Exam lens A reviewer produces too many false positives. The tempting answers add “be conservative” or a confidence filter. The correct move is specific categorical criteria — state exactly what constitutes a finding and what to skip. And when one category is the problem, disable it temporarily to protect trust in the rest while you fix its prompt.

Section 4.2

Apply few-shot prompting to improve output consistency and quality

Task Statement 4.2
🔒
This section is locked.

Complete the knowledge check in Section 4.1 to unlock it.

Go to Section 4.1

When detailed instructions still produce inconsistent output, examples are the fix. Not because the model can’t read — because a worked example carries judgment that prose can’t express, and lets the model generalize to cases you never listed.

What & why

Few-shot examples are the most effective technique for consistently formatted, actionable output when detailed instructions alone fall short. Their real power is in ambiguous cases: an example that shows the reasoning for why one action was chosen over a plausible alternative teaches judgment. And crucially, examples let the model generalize to novel patterns rather than matching only the cases you enumerated — which is why 2–4 well-chosen examples beat a long list of rules. They also reduce hallucination in extraction tasks, where varied document structures otherwise produce invented or empty fields.

Knowledge points
  • Few-shot examples as the most effective technique for consistently formatted, actionable output when instructions alone produce inconsistency.
  • Their role in demonstrating ambiguous-case handling (tool selection for ambiguous requests, branch-level coverage gaps).
  • How examples enable the model to generalize judgment to novel patterns rather than matching only pre-specified cases.
  • Their effectiveness for reducing hallucination in extraction (informal measurements, varied document structures).
In practice
  • Create 2–4 targeted examples for ambiguous scenarios that show the reasoning for why one action was chosen over plausible alternatives.
  • Include examples demonstrating the desired output format (location, issue, severity, suggested fix) for consistency.
  • Provide examples distinguishing acceptable patterns from genuine issues to cut false positives while enabling generalization.
  • Use examples to demonstrate correct handling of varied document structures (inline citations vs bibliographies, methodology sections vs embedded details).
  • Add examples showing correct extraction from varied formats to fix empty/null extraction of required fields.
Anti-patterns the exam tests
  • Writing more detailed instructions when output is inconsistently formatted — examples are the named more-effective technique.
  • Enumerating every case as a rule instead of showing a few examples that generalize — you can’t list every novel pattern.
  • Examples that show only the answer, not the reasoning, for ambiguous cases — the judgment is the point.
Exam lens “Detailed instructions, still inconsistent output” → few-shot examples, every time. For ambiguity, the expected answer is 2–4 examples that show the reasoning, not a longer rule list — because the goal is generalization to cases you didn’t anticipate. Watch for extraction scenarios too: empty/null required fields across varied formats is a few-shot problem.

Section 4.3

Enforce structured output using tool use and JSON schemas

Task Statement 4.3
🔒
This section is locked.

Complete the knowledge check in Section 4.2 to unlock it.

Go to Section 4.2

Asking for JSON and hoping is not a strategy. Tool use with a JSON schema guarantees schema-compliant output — no syntax errors, ever. What it doesn’t guarantee is that the values are right, and that distinction is the most tested idea in this section.

What & why

Tool use with JSON schemas is the most reliable approach for guaranteed schema-compliant structured output — it eliminates JSON syntax errors entirely. The tool_choice setting controls how tools fire: "auto" means the model may return text instead of calling a tool; "any" means it must call a tool but chooses which; forced ({"type":"tool","name":"..."}) means it must call a specific one. The critical caveat: strict schemas eliminate syntax errors but not semantic errors — line items that don’t sum to the total, values in the wrong field. Schema design matters too: fields that may be absent should be optional/nullable, or the model fabricates values to satisfy them.

Knowledge points
  • Tool use with JSON schemas as the most reliable approach for guaranteed schema-compliant output, eliminating JSON syntax errors.
  • tool_choice: "auto" (may return text), "any" (must call a tool, its choice), forced (must call a specific named tool).
  • Strict schemas eliminate syntax errors but not semantic errors (line items that don’t sum, values in wrong fields).
  • Schema design: required vs optional fields, and enum + “other” + detail string patterns for extensible categories.
In practice
  • Define extraction tools with JSON schemas as input parameters and read the structured data from the tool_use response.
  • Set tool_choice: "any" to guarantee structured output when multiple extraction schemas exist and the document type is unknown.
  • Force a specific tool ({"type":"tool","name":"extract_metadata"}) to ensure a particular extraction runs before enrichment steps.
  • Design fields as optional/nullable when source documents may not contain the information, preventing fabrication to satisfy required fields.
  • Add enum values like “unclear” for ambiguous cases and “other” + detail fields for extensible categorization.
  • Include format normalization rules in the prompt alongside strict schemas to handle inconsistent source formatting.
Anti-patterns the exam tests
  • Assuming a strict schema guarantees correct data — it guarantees shape; semantic errors pass straight through.
  • Making a field required when the document may not contain it — this forces the model to fabricate a value.
  • Using tool_choice: "auto" when you need guaranteed structured output — the model may return conversational text instead.
  • Closed enums with no escape hatch — an unanticipated category gets crammed into the nearest wrong bin.
Exam lens Know the three tool_choice modes cold — which guarantees a tool ("any") versus a specific tool (forced) versus nothing ("auto"). Know that schemas fix syntax, not semantics — a scenario where totals don’t add up is a validation problem (4.4), not a schema problem. And “the model invents values for missing data” → make the field nullable, don’t scold it in the prompt.

Section 4.4

Implement validation, retry, and feedback loops for extraction quality

Task Statement 4.4
🔒
This section is locked.

Complete the knowledge check in Section 4.3 to unlock it.

Go to Section 4.3

Section 4.3 got you guaranteed structure. This one catches everything structure can't: values that don't add up, fields in the wrong place, data that simply isn't there. And it teaches the judgment of knowing when a retry will help — and when it's pointless.

What & why

Retry-with-error-feedback means appending the specific validation errors to the prompt on retry, so the model can self-correct against a concrete problem rather than guessing. But retry has a hard limit: it's ineffective when the required information is simply absent from the source document — no amount of retrying conjures data that isn't there. That's the key discrimination: format and structural errors retry successfully; missing information does not. Feedback loops close the circle: tracking a detected_pattern field lets you analyse which code constructs trigger findings that developers dismiss.

Knowledge points
  • Retry-with-error-feedback: appending specific validation errors to the prompt on retry to guide correction.
  • The limits of retry: ineffective when the required information is absent from the source (vs format or structural errors).
  • Feedback loop design: tracking which code constructs trigger findings (detected_pattern) to enable systematic analysis of dismissal patterns.
  • The difference between semantic validation errors (values don't sum, wrong field placement) and schema syntax errors (eliminated by tool use).
In practice
  • Implement follow-up requests including the original document, the failed extraction, and the specific validation errors for self-correction.
  • Identify when retries will be ineffective (information exists only in an external document not provided) versus when they'll succeed (format mismatches, structural output errors).
  • Add detected_pattern fields to structured findings to enable analysis of false-positive patterns when developers dismiss them.
  • Design self-correction validation flows: extract calculated_total alongside stated_total to flag discrepancies; add conflict_detected booleans for inconsistent source data.
Anti-patterns the exam tests
  • Retrying when the information isn't in the document — burns cost and time on something that cannot succeed.
  • Retrying with the same prompt instead of feeding back the specific validation error — the model has nothing new to work with.
  • Assuming tool-use schemas make validation unnecessary — they eliminate syntax errors only; semantic errors need checking.
  • Dismissing false positives without tracking what triggered them — no data to improve the prompt systematically.
Exam lens The discriminating question: can a retry possibly fix this? Format wrong, structure wrong → yes, retry with the error fed back. Information not in the source → no — retry is the wrong answer, route it elsewhere. And the calculated_total vs stated_total pattern is the canonical self-validation trick for catching semantic errors a schema can't.

Section 4.5

Design efficient batch processing strategies

Task Statement 4.5
🔒
This section is locked.

Complete the knowledge check in Section 4.4 to unlock it.

Go to Section 4.4

The Message Batches API halves your cost. It also gives you no latency guarantee and up to a 24-hour window — which makes it perfect for overnight work and completely wrong for anything blocking a merge. Matching the API to the workload is the skill.

What & why

The Message Batches API offers 50% cost savings with an up-to-24-hour processing window and no guaranteed latency SLA. That trade makes it right for non-blocking, latency-tolerant workloads — overnight reports, weekly audits, nightly test generation — and wrong for blocking workflows like pre-merge checks, where someone is waiting. One important constraint: batch does not support multi-turn tool calling within a single request — it can't execute tools mid-request and return results. custom_id fields correlate request/response pairs.

Knowledge points
  • The Message Batches API: 50% cost savings, up to 24-hour processing window, no guaranteed latency SLA.
  • Appropriate for non-blocking, latency-tolerant workloads (overnight reports, weekly audits, nightly test generation); inappropriate for blocking workflows (pre-merge checks).
  • Batch does not support multi-turn tool calling within a single request.
  • custom_id fields for correlating batch request/response pairs.
In practice
  • Match API to latency requirements: synchronous for blocking pre-merge checks, batch for overnight/weekly analysis.
  • Calculate submission frequency from SLA constraints — e.g. 4-hour windows to guarantee a 30-hour SLA given 24-hour batch processing.
  • Handle failures by resubmitting only the failed documents (identified by custom_id) with appropriate modifications (chunking documents that exceeded context limits).
  • Refine prompts on a sample set before batch-processing large volumes to maximize first-pass success and reduce resubmission costs.
Anti-patterns the exam tests
  • Using batch for a blocking workflow — a pre-merge check can't wait up to 24 hours with no SLA.
  • Expecting multi-turn tool calling inside a batch request — not supported.
  • Resubmitting the entire batch when a few documents failed, instead of just the failures by custom_id.
  • Batching 10,000 documents before testing the prompt on a sample — you pay for every failure at scale.
Exam lens The tell is always “is anyone waiting?” Blocking (pre-merge, interactive) → synchronous. Non-blocking (overnight, weekly) → batch, for the 50% saving. Expect arithmetic too: given a 24-hour window and an SLA, compute the submission cadence. And know the no-multi-turn-tools constraint — it rules batch out for agentic workloads.

Section 4.6

Design multi-instance and multi-pass review architectures

Task Statement 4.6
🔒
This section is locked.

Complete the knowledge check in Section 4.5 to unlock it.

Go to Section 4.5

A model that just wrote the code carries its own reasoning forward — it's already convinced. Asking it to review its own work, however firmly, doesn't undo that. An independent instance without the generation context catches what self-review can't.

What & why

Self-review is limited by construction: the model retains reasoning context from generation, making it less likely to question its own decisions in the same session. Independent review instances — without prior reasoning context — are more effective at catching subtle issues than self-review instructions or extended thinking. The other half is multi-pass review: splitting large reviews into per-file local analysis passes plus separate cross-file integration passes, which avoids attention dilution and the contradictory findings that come from trying to hold everything at once.

Knowledge points
  • Self-review limitations: the model retains reasoning context from generation, so it's less likely to question its own decisions in the same session.
  • Independent review instances (without prior reasoning context) are more effective at catching subtle issues than self-review instructions or extended thinking.
  • Multi-pass review: per-file local analysis passes plus cross-file integration passes, avoiding attention dilution and contradictory findings.
In practice
  • Use a second independent Claude instance to review generated code without the generator's reasoning context.
  • Split large multi-file reviews into focused per-file passes for local issues plus separate integration passes for cross-file data flow.
  • Run verification passes where the model self-reports confidence alongside each finding, enabling calibrated review routing.
Anti-patterns the exam tests
  • Asking the generating session to review its own output — it's the named limitation, and no instruction fixes it.
  • Reaching for extended thinking to improve self-review — explicitly less effective than an independent instance.
  • One undivided pass over a large multi-file change — attention dilutes, findings contradict.
Exam lens “The reviewer misses issues in code it generated” → an independent instance, not a sterner instruction and not extended thinking. Both of those are written as distractors here. For large reviews, the expected architecture is per-file passes + a cross-file integration pass. Confidence self-reporting shows up as the routing mechanism.

Module 4 · Wrap-up

What this domain comes down to

Prompt Engineering & Structured Output is about precision and verification: explicit criteria instead of vague exhortation (4.1), examples that carry judgment prose can't (4.2), schemas that guarantee shape via tool use (4.3), validation that catches what shape can't (4.4), the right API for the latency you need (4.5), and review architectures that don't let a model mark its own homework (4.6).

One thread runs through the anti-patterns: guarantees are narrower than they look. A strict schema guarantees syntax, not semantics. A retry fixes format, not absent data. “Be conservative” guarantees nothing at all. And a model reviewing its own work brings its own conclusions with it. Know exactly what each mechanism guarantees — and put a check on everything it doesn't.
