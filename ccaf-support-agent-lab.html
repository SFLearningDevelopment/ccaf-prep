Build a support agent
that knows when to act.

A single hands-on capstone: a customer-support resolution agent built on the Claude Agent SDK, reaching into your backend through four custom MCP tools. It resolves the clear cases — returns, billing questions, account fixes — and escalates the genuinely ambiguous ones. Target: 80%+ first-contact resolution without ever guessing at an action that moves money.

Before you start

You should be comfortable in a terminal and reading code — this lab assumes the developer experience the exam itself assumes (roughly 6+ months building with Claude). It does not stop to explain package managers or async functions.

  • Python 3.10+ or Node 18+ — the Agent SDK ships for both. Examples lead in Python; the shape is identical in TypeScript.
  • An Anthropic API key exported as ANTHROPIC_API_KEY. (Bedrock or Vertex auth also works.)
  • No real backend needed — you’ll build the four tools against a small in-memory fake dataset (a few customers and orders). That’s deliberate: it keeps process_refund safe to run while you learn to guard it.
Verify before you build The Agent SDK, its MCP tooling, and the permission system have all moved recently and continue to. Package names, the permission-evaluation order, and a few option fields below carry real shelf-life. Wherever you see a Verify note, check it against the current docs at code.claude.com/docs and platform.claude.com/docs. The shape of each pattern is stable; exact identifiers drift.
Orientation

What you’re building, before you build it

By the end of this lab you’ll have one working program: a support agent that takes a customer message (“I want a refund for order 4471, it arrived broken”), identifies the customer, pulls up the order, decides whether it can resolve the request itself, and either acts — issuing a guarded refund — or escalates to a human with the context already gathered. Every stage adds one capability to that same agent. You are not building seven disconnected exercises; you are building one agent in seven passes.

The one thing you’re building

A customer-support resolution agent. It reads a high-ambiguity request, reaches into backend systems through four custom MCP tools (get_customer, lookup_order, process_refund, escalate_to_human), resolves what it safely can, and hands off what it can’t — aiming for 80%+ first-contact resolution.

input → a customer request output → resolution or escalation guardrail → refunds never guessed

How the seven stages connect

Read the diagram as a decision flow. The resolution loop (Stage 1) is the engine; the four MCP tools (Stages 2–3) are how it reaches your backend; a guard (Stage 4) sits in front of the one action that moves money; the decision (Stage 5) routes each request to resolve or escalate; context management (Stage 6) keeps the agent coherent across a multi-turn conversation; and measurement (Stage 7) proves it hits the target.

Don’t worry if a box uses a word you don’t know yet — the glossary below defines every term in the diagram, and each stage explains its own box in full when you reach it.

The terms this lab uses

These come up across multiple stages. Skim them now; you don’t need to memorize them — they’ll make sense as you build.

Resolution loop
The cycle the SDK runs for you: send the model the customer request, let it call a tool (look up the customer, fetch an order), feed the result back, repeat — until it can resolve or decides to escalate. The agent loop, in support terms.
Claude Agent SDK
The library that runs the loop, tools, permissions, and context management for you, from Python or TypeScript. The renamed successor to the “Claude Code SDK” — old names claude-code-sdk / ClaudeCodeOptions are stale.
MCP (Model Context Protocol)
The open protocol that connects the agent to external systems. Here you use the SDK’s in-process MCP server to expose four functions as tools — no separate server process.
Custom tool
One of your own functions the agent can call, defined with the @tool decorator (Python) or tool() helper (TS): a name, a description, an input schema, and a handler. Your four are get_customer, lookup_order, process_refund, escalate_to_human.
Read tool vs. action tool
A distinction this lab leans on. Read tools (get_customer, lookup_order) only fetch — safe to call freely. Action tools (process_refund) change the world — they need a guard.
Session
One continuous run of the agent with memory — here, one customer conversation. Multi-turn (“actually, also order 4472”) means resuming the same session so the agent keeps the context.
Context (window)
Everything the model can see at once — the conversation plus every tool result so far. It’s finite, so a long back-and-forth can fill it; keeping it small and relevant is Stage 6.
Permission gate (canUseTool / hooks)
The SDK’s runtime checkpoint before a tool runs. You use it to stop process_refund from executing unless the request passes your rules — the core of Stage 4’s guard.
Idempotency
The property that repeating the same action produces the same result — so a retried refund doesn’t pay out twice. Enforced with an idempotency key the action tool remembers.
Escalation / human handoff
Handing a request to a person instead of resolving it — on a defined trigger (policy gap, explicit request, can’t progress) or when confidence is too low. The escalate_to_human tool, plus the judgment for when to call it.
First-contact resolution (FCR)
The share of requests fully resolved in one interaction without handing off. The lab’s 80% target — measured against test tickets, not assumed.
Error category / retryable flag
Structure in a tool’s error response — a category (not_found, rate_limited…) and whether retrying could help — so the loop can decide to retry, fall back, or escalate instead of treating every failure the same.
Before Stage 1
  • You can state, in one sentence, what the finished agent does end to end.
  • You can trace a single request through the diagram: loop → read tools → decision → refund or escalate.
  • You can name which of the four tools is the dangerous one, and why.
Stage 01

Stand up the resolution loop

Agentic Architecture & Orchestration · 27%

Everything rests on one loop: send the customer’s request to the model, run the tools it asks for, feed results back, repeat until it can resolve or escalate. Build that first, then make it hold a multi-turn conversation.

“I want a refund for order 4471 — it showed up cracked.” The agent needs to find the customer, pull the order, judge the request, and act — possibly over several turns.

1a · Install and make one query

The Claude Agent SDK is the renamed successor to the old “Claude Code SDK.” If a tutorial imports claude-code-sdk or ClaudeCodeOptions, it’s stale — the code won’t run.

Install
# Python 3.10+ pip install claude-agent-sdk # TypeScript (bundles the Claude Code binary — nothing else to install) npm install @anthropic-ai/claude-agent-sdk
The loop, with a support system prompt (Python)
import asyncio from claude_agent_sdk import query, ClaudeAgentOptions SYSTEM = """You are a customer support resolution agent. Resolve returns, billing questions, and account issues when you can do so confidently and within policy. Look up the customer and their order before deciding. If you cannot resolve confidently, escalate — do not guess.""" async def main(): async for message in query( prompt="Customer says: refund for order 4471, arrived cracked.", options=ClaudeAgentOptions( system_prompt=SYSTEM, max_turns=10, # circuit breaker — never loop forever ), ): if hasattr(message, "result"): print(message.result) asyncio.run(main())

You did not hand-write the loop. query() runs the full model–tool–result cycle and streams every message back, ending with a result message. Right now the agent has no tools — it can only talk. Stage 2 gives it the four it needs.

1b · Cap the run and handle failure

The SDK ships no default turn limit, budget cap, or retry logic. For a customer-facing agent, max_turns is your circuit breaker and a budget cap is your cost ceiling. Wrap the loop and branch on the result’s error subtype (rate_limit, overloaded, authentication_failed, server_error) rather than treating every failure alike.

options=ClaudeAgentOptions( system_prompt=SYSTEM, max_turns=10, # set a per-session budget cap in production — support traffic is high-volume )

1c · Hold the conversation (sessions)

Support is multi-turn: the customer replies, corrects, adds a second order. Each query() starts fresh, so capture the session_id from the init message and resume it on the next turn — that’s how the agent remembers it already looked up this customer.

# Turn 1: grab session_id from the init message. # Turn 2 ("actually, also order 4472"): resume it. options=ClaudeAgentOptions(system_prompt=SYSTEM, resume=session_id)
For a continuous conversation, ClaudeSDKClient keeps one session open across many query() calls — often cleaner for chat than resuming by ID each turn. Either works; pick one and be consistent.
Exam lens Scenarios test whether you know the SDK runs the loop for you (versus the raw Messages API, where you write the tool_use plumbing), and that an agent without an explicit max_turns / budget cap is a production hazard. Session continuity — remembering earlier turns — is the other reliable theme here.
Verify The SDK is pre-1.0 and moving. Confirm the import path, ClaudeAgentOptions fields, system_prompt, and the session-resume shape against code.claude.com/docs/en/agent-sdk before relying on them.
Stage 1 done when
  • Your agent responds to a customer message over multiple turns, with a system prompt that tells it to resolve-or-escalate.
  • A follow-up turn resumes the same session with full context.
  • max_turns is set, and you handle at least one error subtype deliberately.
Stage 02

Build the four MCP tools

Tool Design & MCP Integration · 18%

The agent reaches your backend through four custom tools. You’ll define them as an in-process MCP server — ordinary functions the SDK exposes to the model. Back them with a small fake dataset so the whole thing runs locally and process_refund is harmless while you learn to guard it.

2a · A fake backend to call

Two dictionaries stand in for your real systems. Everything the tools return comes from here.

CUSTOMERS = { "cust_88": {"id": "cust_88", "name": "Rae Ortiz", "tier": "standard", "email": "rae@example.com"}, } ORDERS = { "4471": {"id": "4471", "customer_id": "cust_88", "amount": 42.00, "status": "delivered", "refundable": True}, } REFUNDS_DONE = {} # idempotency ledger — Stage 4 uses this

2b · Define the tools as an in-process MCP server

Each tool is four parts: a name the model calls, a description that tells it when to use it, an input schema, and a handler. Wrap them with create_sdk_mcp_server — it runs inside your process, no subprocess.

import json from claude_agent_sdk import tool, create_sdk_mcp_server @tool("get_customer", "Fetch a customer record by customer_id.", {"customer_id": str}) async def get_customer(args): c = CUSTOMERS.get(args["customer_id"]) if not c: return {"content": [{"type": "text", "text": '{"error":{"category":"not_found","retryable":false}}'}]} return {"content": [{"type": "text", "text": json.dumps(c)}]} @tool("lookup_order", "Fetch one order by order_id, including amount and refundable flag.", {"order_id": str}) async def lookup_order(args): o = ORDERS.get(args["order_id"]) if not o: return {"content": [{"type": "text", "text": '{"error":{"category":"not_found","retryable":false}}'}]} return {"content": [{"type": "text", "text": json.dumps(o)}]} @tool("process_refund", "Issue a refund for an order. Moves money — only call after verifying " "the order is refundable and within policy. Requires order_id and amount.", {"order_id": str, "amount": float, "idempotency_key": str}) async def process_refund(args): # Stage 4 adds the real guard. For now, record it against the ledger. key = args["idempotency_key"] if key in REFUNDS_DONE: return {"content": [{"type": "text", "text": json.dumps(REFUNDS_DONE[key])}]} # idempotent replay receipt = {"refunded": args["amount"], "order_id": args["order_id"], "key": key} REFUNDS_DONE[key] = receipt return {"content": [{"type": "text", "text": json.dumps(receipt)}]} @tool("escalate_to_human", "Hand this case to a human agent. Call when the request is outside policy, " "explicitly requested, or you cannot resolve it confidently. Include a reason " "and the context gathered so far.", {"reason": str, "summary": str}) async def escalate_to_human(args): ticket = {"escalated": True, "reason": args["reason"], "summary": args["summary"]} return {"content": [{"type": "text", "text": json.dumps(ticket)}]} support = create_sdk_mcp_server(name="support", version="1.0.0", tools=[get_customer, lookup_order, process_refund, escalate_to_human])

2c · Give the agent the tools

Pass the server in mcp_servers. Tool names follow the pattern mcp__{server}__{tool}. Listing them in allowed_tools pre-approves the safe ones so they run without a prompt — note we deliberately leave process_refund out so it falls through to the guard you build in Stage 4.

options = ClaudeAgentOptions( system_prompt=SYSTEM, mcp_servers={"support": support}, allowed_tools=[ "mcp__support__get_customer", "mcp__support__lookup_order", "mcp__support__escalate_to_human", # process_refund intentionally omitted — gated in Stage 4 ], max_turns=10, )
Subtlety worth knowing: allowed_tools controls pre-approval, not availability. A tool the agent can see but that isn’t pre-approved still gets offered to the model — it just has to pass the permission step when called. That’s exactly the behavior you want for process_refund.
Exam lens Know the four parts of a tool and that custom tools are in-process MCP servers via create_sdk_mcp_server. And know the availability-vs-permission distinction: allowed_tools pre-approves; it does not decide what the model can see. Distractors blur those two layers.
Verify The @tool signature, create_sdk_mcp_server, and the mcp__server__tool naming are current, but the schema shorthand ({"customer_id": str} vs. a full JSON Schema) and allowed_tools semantics can shift. Confirm against the custom-tools guide before relying on them.
Stage 2 done when
  • All four tools are defined and wrapped in one in-process MCP server.
  • The agent calls get_customer and lookup_order on its own to gather context.
  • process_refund is deliberately not pre-approved — ready for its guard.
Stage 03

Differentiate & error-proof the tools

Tool Design & MCP Integration · 18%

The agent only picks the right tool if your descriptions make the choice obvious, and only recovers from failure if your errors tell it what went wrong. Sharpen both, then try to trip the agent up on purpose.

3a · Descriptions that draw a boundary

In a support flow, the confusable pair is get_customer vs. lookup_order — both “look something up.” State what each is for and, crucially, when not to reach for it.

# get_customer "Fetch a CUSTOMER record (name, tier, email) by customer_id. " "Use to identify who you're helping. Do NOT use to look up an order — " "it has no order data; use lookup_order for that." # lookup_order "Fetch ONE ORDER (amount, status, refundable) by order_id. " "Use to check an order before acting on it. Do NOT use to find a customer's " "details — use get_customer for that."

3b · Structured errors the loop can act on

You already returned {"error":{"category":...,"retryable":...}} in Stage 2. That structure is the point: it lets the agent branch instead of getting stuck. A crucial SDK fact — a handler that returns an error keeps the loop alive; the model reads it and adapts. A handler that throws an uncaught exception is caught by the SDK and also returned as an error result, but you lose control of the message. Prefer returning structured errors.

# category drives the agent's next move: # not_found → stop, tell the customer, maybe escalate # invalid_input → fix the arguments and retry # rate_limited → wait and retry # upstream_down → escalate; don't spin

3c · Try to confuse it

Feed ambiguous requests and watch which tool the agent picks:

  • “Look up 4471” → a bare number reads as an order → should pick lookup_order.
  • “Who is cust_88?” → should pick get_customer.
  • “Check on Rae’s cracked order” → genuinely needs both, in sequence: customer first, then order. A good tool set makes the agent chain them; a poor one stalls.

When it routes wrong, fix the description, not the model — tighten the “do NOT use when” clause and re-test.

Exam lens This domain is judgment about tool boundaries and structured errors. Expect two overlapping tools and a “why did it mis-select” question — the fix sharpens the boundary with negative guidance. And know why a category + retryable flag beats a bare error string: it turns a dead end into a decision.
Stage 3 done when
  • Your two read tools have descriptions with explicit “do NOT use when” boundaries.
  • Every tool returns errors as a category + retryable flag, and the agent visibly adapts to one.
  • You’ve run three ambiguous requests and logged which tool got picked and why.
Stage 04

Guard the money-moving action

Agentic Architecture & Orchestration · 27%

Three of your tools only read. One moves money. process_refund is where a model mistake, a prompt injection, or a retry becomes a real financial event — so it never runs on the model’s say-so alone. You put a deterministic gate in front of it.

The agent decides a refund is warranted and calls process_refund(order_id="4471", amount=4200.00) — two decimal places off. Without a guard, that pays out $4,200 on a $42 order.

4a · Gate the call with canUseTool

The SDK evaluates every tool call through a fixed order: PreToolUse hook → deny rules → allow rules → permission mode → can_use_tool callback → PostToolUse hook. Because you left process_refund out of allowed_tools, its calls fall through to your can_use_tool callback — your runtime decision point. Approve only what passes policy; deny the rest with a message the model reads and adapts to.

from claude_agent_sdk.types import PermissionResultAllow, PermissionResultDeny REFUND_CAP = 200.00 # auto-approve ceiling; above this, a human decides async def gate(tool_name, args, context): if tool_name == "mcp__support__process_refund": order = ORDERS.get(args.get("order_id", "")) if not order or not order["refundable"]: return PermissionResultDeny(message="Order not refundable — escalate instead.") if args["amount"] > order["amount"]: return PermissionResultDeny(message="Refund exceeds order amount. Recheck.") if args["amount"] > REFUND_CAP: return PermissionResultDeny(message="Above auto-approve cap — use escalate_to_human.") return PermissionResultAllow() # within policy → allow return PermissionResultAllow() # read tools & escalate: fine options = ClaudeAgentOptions( system_prompt=SYSTEM, mcp_servers={"support": support}, allowed_tools=["mcp__support__get_customer", "mcp__support__lookup_order", "mcp__support__escalate_to_human"], can_use_tool=gate, max_turns=10, )

Now the $4,200 slip is caught by the amount > order["amount"] check and denied — the model reads the denial message and corrects or escalates, and no money moves.

4b · For guarantees that must always hold, use a hook

A can_use_tool gate is your main line, but note one sharp edge from the evaluation order: a tool pre-approved by allowed_tools or a permissive mode skips the callback. If a refund guarantee must hold in every mode — even a misconfigured one — put it in a PreToolUse hook, which runs first and whose deny beats everything, including bypass mode. Belt and braces for the one action that moves money.

4c · Idempotency: never pay twice

Retries happen — a timeout, a resumed session, a double-click upstream. Your tool already keys each refund and replays the stored receipt instead of issuing a second one (the REFUNDS_DONE ledger from Stage 2). Make the agent pass a stable idempotency_key per request so a retry lands on the same key. This is the difference between a robust action tool and a lawsuit.

Exam lens This is heavily tested. Know the permission evaluation order cold, and the two footguns: allowed_tools does not constrain bypassPermissions, and a pre-approved tool skips can_use_tool. Hard guarantees belong in deny rules or PreToolUse hooks because deny always wins. Idempotency and caps are the canonical guardrails for a money-moving tool. “Trust the model to only refund the right amount” is always the wrong answer.
Verify PermissionResultAllow/Deny, the can_use_tool signature, and the exact evaluation order are current but version-sensitive (the TS SDK even warns when a callback is unreachable). Confirm against the permissions guide at code.claude.com/docs/en/agent-sdk/permissions.
Stage 4 done when
  • A refund above the cap, above the order amount, or on a non-refundable order is denied — and the agent adapts.
  • You can explain why a hard guarantee goes in a hook, not just the callback.
  • A retried refund with the same key pays out once, not twice.
Stage 05

Decide: resolve or escalate

Context Management & Reliability · 15%

The target is 80%+ first-contact resolution — which is not “resolve everything.” It’s resolving the clear cases confidently and escalating the genuinely ambiguous ones, cleanly, with context attached. Get this line right and the metric follows.

5a · When to escalate vs. resolve

Escalate on a defined trigger, not a hunch. The three that matter:

  • Policy gap — nothing in the agent’s rules covers the case (a refund past the window, a dispute on a closed account). No basis to decide → escalate.
  • Explicit request — the customer asks for a human, or a rule requires sign-off (refunds over the cap, from Stage 4).
  • Inability to progress — a tool keeps returning upstream_down, the account data conflicts, the agent can’t verify the claim. Don’t spin → escalate.

Everything else — the in-policy refund, the simple status question, the address fix — it resolves autonomously. That’s the 80%.

5b · Escalate with context, not a cold handoff

A good escalation isn’t “I can’t help” — it’s a handoff a human can act on in seconds. Your escalate_to_human tool already takes a reason and a summary; require the summary to carry what the agent gathered: customer, order, what was attempted, why it stopped.

escalate_to_human( reason="refund_past_window", summary="Rae Ortiz (cust_88), order 4471 ($42, delivered 40 days ago). " "Requests refund for damage. Past 30-day window — needs human judgment." )

5c · Confidence-based routing

Where the agent can attach a confidence to its own conclusion, route on it: high confidence and in-policy → resolve; low confidence or out-of-policy → escalate. The middle band is the whole point — not confident enough to act, so a human decides. Routing everything to a human defeats the agent; routing nothing is how bad refunds ship.

Exam lens Pure judgment, and scenario-heavy. A situation is described; you pick resolve or escalate. Map to the triggers: policy gap / explicit request / can’t progress → escalate; clear and in-policy → resolve. Beware answers that maximize the FCR metric by never escalating — the exam treats gaming the metric as the wrong call.
Stage 5 done when
  • The agent resolves clear in-policy cases and escalates on each of the three triggers.
  • Every escalation carries a summary a human could act on without re-asking the customer.
  • You can articulate why 80% FCR is a ceiling on escalation, not a floor on resolution.
Stage 06

Manage context across the conversation

Context Management & Reliability · 15%

A real support conversation runs long: several turns, repeated lookups, verbose tool payloads. The context window fills and the agent drifts — forgets the order it already fetched, re-asks what the customer said. Keep the working context small and organized.

6a · Extract facts from verbose tool output

A full customer record or order history can be a wall of fields the agent mostly doesn’t need. Carry the distilled facts forward, not the raw dump.

# Instead of the whole customer + order JSON, carry what the decision needs: { "customer": "cust_88 (standard)", "order": "4471 $42 delivered refundable" }

6b · A scratchpad for the case

For a multi-turn case, keep running state outside the context window — a short case note the agent updates: who, what they want, what’s been checked, what’s pending. A resumed session reads the note instead of re-deriving everything. State lives outside the window; the window holds only what the current turn needs.

6c · Delegate a noisy sub-task

If resolving a billing dispute means reading a long transaction history, spin that into a subagent: it reads the history in its own context and returns a three-line summary. The parent pays for the summary, not the whole history — the second reason subagents exist, alongside specialization.

The SDK compacts automatically as the limit nears and re-reads your system prompt after compaction — but designing for small context beats relying on compaction, especially in a high-volume support agent where every extra token is paid per conversation.
Exam lens Scenarios describe an agent degrading over a long conversation; you pick the fix. The answers cluster around fact extraction (carry distilled facts, not raw output), external state (scratchpad over in-context memory), and delegation for context isolation. Know why each shrinks the working set.
Stage 6 done when
  • Verbose tool output is distilled to the facts the decision needs before moving on.
  • A case note holds state across turns / a resumed session.
  • At least one noisy sub-task is delegated to keep the main context small.
Stage 07

Measure first-contact resolution

Reliability in practice

“80%+ first-contact resolution” is only real if you measure it. Build a small set of test tickets, define what “resolved” means, run the agent, and compute the rate. Then tune — without gaming the number by suppressing escalations you should be making.

7a · Define “resolved on first contact”

Pin it down before measuring. A reasonable definition for this agent: the request reached a correct terminal outcome in one conversation, with no handoff — either a valid in-policy refund was issued, or the correct answer was given. An escalation is not a resolution — but escalating when you should is a correct outcome, not a failure. Those are different axes: track resolution rate and escalation-correctness.

7b · A test set with a known right answer

Write tickets that span the real distribution — clear resolvables and genuine escalations — each labeled with what should happen.

TEST_TICKETS = [ {"msg": "Refund order 4471, arrived cracked.", "expect": "refund"}, {"msg": "What's the status of order 4471?", "expect": "answer"}, {"msg": "Refund my order from 6 months ago.", "expect": "escalate"}, # past window {"msg": "I want to speak to a person about a dispute.", "expect": "escalate"}, # explicit {"msg": "Refund order 4471 for $4200.", "expect": "deny_then_correct"}, ]

Run each through the agent, capture the terminal outcome, and score it against expect. Your unit tests cover the tools as plain functions (no model needed); this is the end-to-end harness — pin a model for repeatability and keep max_turns tight.

7c · Compute FCR, then tune the honest way

First-contact resolution = correct in-policy resolutions ÷ total tickets that should resolve without a human. If you’re under 80%, the fix is one of: sharper tool descriptions (Stage 3) so the agent gathers the right facts, a clearer resolve/escalate line (Stage 5), or a system prompt that better states policy. The dishonest fix — suppressing escalations to inflate the rate — trades a metric for real refunds gone wrong. Track escalation-correctness alongside FCR precisely so that gaming shows up.

Exam lens The exam frames reliability as measurable: evals over vibes, a labeled test set, a metric with a definition. And it consistently rewards the honest reading of a target — 80% FCR is achieved by resolving what’s resolvable, not by refusing to escalate. Expect a scenario where raising FCR the wrong way is the tempting distractor.
Stage 7 done when
  • “Resolved on first contact” has a written definition, separate from escalation-correctness.
  • A labeled test set runs end to end and produces an FCR number.
  • You’ve tuned toward 80%+ without suppressing warranted escalations — and can show both metrics moving.
Wrap-up

What you built, and how it maps

You now have one working agent that exercises the scenario’s three primary domains: a resolution loop with sessions and subagents (Agentic Architecture, 27%); four custom MCP tools with sharp descriptions, structured errors, and a permission-gated money-moving action (Tool & MCP, 18%, plus the guardrail judgment that lives in Agentic Architecture); and context management with confidence-based escalation and a measured 80% first-contact-resolution target (Context & Reliability, 15%). Completing it, you should be able to answer any question the scenario raises — how the loop works, why process_refund is gated where it is, when to escalate, and how the target is measured rather than assumed.

The exam is 60 scenario questions in 120 minutes, and it rewards judgment over recall — the wrong answers are written to tempt people with partial knowledge. Having actually built the guard, the escalation line, and the measurement harness is the difference between recognizing the right call and guessing.
