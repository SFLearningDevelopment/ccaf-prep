Before you start

You should be comfortable in a terminal and reading code — this lab assumes the developer experience the exam itself assumes (roughly 6+ months building with Claude). It does not stop to explain async functions or git.

  • Python 3.10+ or Node 18+ — the Agent SDK ships for both. Examples lead in Python; the shape is identical in TypeScript.
  • An Anthropic API key exported as ANTHROPIC_API_KEY. Bedrock/Vertex/Foundry auth also works.
  • A real repo you don’t know well — an open-source project you’ve never read, or a legacy service at work. The point is to let the agent explore something genuinely unfamiliar. Run it in a container or VM once you grant write/Bash access.
Verify before you build The Agent SDK, its built-in tool set, permission modes, and MCP config shapes have all moved recently and continue to. Package names, tool names, the allowed_tools semantics, and the MCP server config union carry real shelf-life. Wherever you see a Verify note, check it against the current docs at code.claude.com/docs. The shape of each pattern is stable; exact identifiers drift.
Orientation

What you’re building, before you build it

By the end of this lab you’ll have one working program: an agent you point at a repository that helps a developer get productive fast — it explores unfamiliar code and explains how it fits together, generates boilerplate to your conventions, automates a repetitive chore, and reaches into your own systems through an MCP server. Each stage adds one capability, and — just as important — widens what the agent is allowed to touch, from read-only to trusted. You are not building seven disconnected exercises; you are building one agent in seven passes.

The one thing you’re building

A developer-productivity agent on the Agent SDK. It uses the built-in tools — Read, Write, Bash, Grep, Glob — to explore, explain, generate, and automate, and integrates an MCP server to reach the systems that live behind your own APIs. You control exactly what it can do at each step.

input → a repo + a task output → understanding, code, automation control → least privilege, widened deliberately
Not the same as the Claude Code workflow lab. That lab was about configuring the Claude Code product (CLAUDE.md, slash commands, plan mode). This lab is about building your own agent with the SDK — the library underneath Claude Code — and driving it from your code. Same engine, different seat: there, you’re the user; here, you’re the author.

How the seven stages connect

Read the diagram as one agent whose reach grows stage by stage. It starts read-only (Stages 1–3): explore and understand, touching nothing. Then you grant write (Stage 4) to generate, and Bash (Stage 5) to automate — each behind a guard. An MCP server (Stage 6) extends its reach to your own systems, and configuration (Stage 7) makes the whole thing reusable across the team. The dashed band tracks the widening trust boundary.

Don’t worry if a box uses a word you don’t know yet — the glossary below defines every term in the diagram, and each stage explains its own box in full when you reach it.

The terms this lab uses

These come up across multiple stages. Skim them now; you don’t need to memorize them — they’ll make sense as you build.

Claude Agent SDK
The library that runs the agent loop, built-in tools, permissions, and context management for you, from Python or TypeScript — the same engine that powers Claude Code, driven by your code. Renamed from “Claude Code SDK”; old names claude-code-sdk / ClaudeCodeOptions are stale.
Built-in tools
Tools the SDK ships with, no implementation needed: Read, Write, Edit (files), Bash (shell), Glob (find files by pattern), Grep (search contents), plus WebSearch/WebFetch and Agent (subagents).
allowed_tools
The list of tools the agent may use — and the agent’s security boundary. A tool not on the list can’t be called. A read-only agent lists only Read, Glob, Grep.
Least privilege
The governing principle of this lab: start the agent with the minimum tools for the job and grant more only when a stage needs it. A read-only agent can analyze anything and damage nothing.
Glob & Grep
The two exploration tools. Glob finds files by name pattern (**/*.py); Grep searches file contents by regex. Together they’re how the agent navigates a codebase it’s never seen.
Plan mode / read-only
A permission posture where the agent investigates and proposes but changes nothing. Ideal for exploring an unfamiliar or legacy codebase safely before granting any write access.
Permission mode
How the agent handles actions: default (asks on risk), acceptEdits (auto-approves edits), plan (read-only), plus a runtime can_use_tool callback for custom gating. The lever you widen as trust grows.
Guard (can_use_tool / hook)
A runtime checkpoint before a tool runs — used here to gate the risky grants (a Bash command, a write outside the working directory). Deny what shouldn’t run; allow the rest.
MCP (Model Context Protocol)
The protocol that connects the agent to systems beyond the filesystem — a database, an issue tracker, an internal API. External servers connect by command or URL; your own domain logic can be an in-process server.
mcp__server__tool
How MCP tool names are qualified: mcp__<server>__<tool>. That full string is what goes in allowed_tools to permit the tool.
Context (window) / scratchpad
What the agent holds at once. Exploring a large legacy system can flood it, so distilled findings and an on-disk scratchpad keep the working context lean — the Stage 3 discipline.
Project configuration
Filesystem settings (CLAUDE.md, settings) the SDK can load with setting_sources=["project"], so the agent inherits your team’s conventions and everyone gets the same behavior.
Before Stage 1
  • You can state, in one sentence, what the finished agent does end to end.
  • You can trace the widening trust: read-only → write → Bash → your systems via MCP.
  • You can explain how this lab differs from configuring the Claude Code product.
Stage 01

Stand up a least-privilege agent

Agentic Architecture & Orchestration · 27%

Every capability this agent gains, it earns. Start with the safest possible version — one that can read and search but not change anything — then widen deliberately in later stages. Least privilege isn’t caution for its own sake; it’s the design that lets you trust the agent in a real repo.

You point the agent at a service you’ve never seen. Before it writes a line or runs a command, you want it to be able to explore and explain — and to be structurally incapable of breaking anything.

1a · Install and make a read-only agent

The Claude Agent SDK is the renamed successor to the old “Claude Code SDK.” If a tutorial imports claude-code-sdk or ClaudeCodeOptions, it’s stale.

Install
# Python 3.10+ pip install claude-agent-sdk # TypeScript (bundles the Claude Code binary — nothing else to install) npm install @anthropic-ai/claude-agent-sdk
A read-only agent, pointed at a repo (Python)
import asyncio from claude_agent_sdk import query, ClaudeAgentOptions async def main(): async for message in query( prompt="Summarize what this service does and how it's structured.", options=ClaudeAgentOptions( cwd="/path/to/unfamiliar/repo", # point it at the codebase allowed_tools=["Read", "Glob", "Grep"], # read-only: analyze, never modify permission_mode="plan", # extra belt: read-only posture max_turns=30, ), ): if hasattr(message, "result"): print(message.result) asyncio.run(main())

The agent can now read files, glob for patterns, and grep contents — and nothing else. It literally cannot write or run a command, because those tools aren’t on the list.

1b · allowed_tools is the security boundary

This is the idea to internalize: allowed_tools isn’t just convenience, it’s the boundary. A tool not on the list cannot be called, full stop. So you widen it one grant at a time as each stage needs it — Write in Stage 4, Bash in Stage 5 — rather than granting everything up front and hoping the agent behaves.

1c · Cap the run

Set max_turns as a circuit breaker so exploration can’t loop forever, and read token usage from the result to keep an eye on cost. Once you grant write or Bash access in later stages, run the agent in a container or VM — the environment becomes part of the safety boundary.

Exam lens Know that allowed_tools is the security boundary, and that least privilege is the right default — start read-only, widen as needed. “Give the agent all tools so it’s flexible” is the wrong instinct the exam probes. Know the built-in tool set by name and which combination makes an agent read-only (Read, Glob, Grep).
Verify Package name, ClaudeAgentOptions fields, cwd, and permission_mode values are current but the SDK is pre-1.0 and moving. Confirm against code.claude.com/docs/en/agent-sdk.
Stage 1 done when
  • Your agent runs against a real repo and returns a summary using only read tools.
  • It’s structurally read-only — no Write, Edit, or Bash on the list.
  • max_turns is set and you can read token usage from the result.
Stage 02

Explore an unfamiliar codebase

Tool Design & MCP Integration · 18%

The first real job: help a developer get oriented in code they’ve never seen. This is what Glob and Grep are for — the agent navigates by pattern and search, the same way you would, but faster and more thoroughly.

2a · The exploration tools, and what each is for

Two tools do the navigating, and knowing which is which matters:

  • Glob finds files by name pattern — **/*.py, src/**/handlers/*. Use it to map the shape of the project: where the entry points, tests, and config live.
  • Grep searches file contents by regex — where is UserSession defined, every call site of process_payment. Use it to trace how things connect.
  • Read pulls a specific file in once Glob/Grep have pointed you at it.

The agent chains them on its own: glob to find, grep to locate, read to understand.

2b · Explore in plan mode, safely

Exploration should never risk a change. Keep the agent in the read-only posture from Stage 1 — permission_mode="plan" plus a read-only tool list — so it can crawl the whole repo, follow leads, and build understanding without any possibility of a write. Ask it real orientation questions:

# Orientation prompts that exercise Glob → Grep → Read: "What are the entry points, and what does each do?" "Trace how a request flows from the API layer to the database." "Where is authentication handled, and what pattern does it use?"

2c · Return findings, not raw files

A large repo has more content than fits usefully in context. Prompt the agent to return structured findings — a map of the architecture, the key modules, the entry points — rather than dumping files back at you. This is the exploration equivalent of a good summary: the conclusion, not the crawl. (Stage 3 leans on this hard for legacy systems.)

Exam lens Know the agentic-search tools and their division of labor: Glob by filename pattern, Grep by content regex, Read for the specific file. Scenarios ask which tool fits a task (“find every call site” → Grep; “list all test files” → Glob). Exploring read-only before acting is the safe, expected posture.
Verify Tool names and the read-only combination are stable, but permission_mode values and how Glob/Grep accept patterns can shift. Confirm against the SDK tools reference.
Stage 2 done when
  • The agent maps an unfamiliar repo’s structure using Glob, Grep, and Read on its own.
  • It answers a real “how does X work here?” question by tracing through the code.
  • It returns structured findings, not a pile of raw files.
Stage 03

Understand a legacy system

Context Management & Reliability · 15%

A legacy system is exploration at scale: too much code to hold in context at once, sprawling and interconnected. The challenge stops being “can the agent read it” and becomes “can it stay coherent while reading a lot.” That’s a context-management problem.

You ask: “How is error handling done across this 200-file service, and is it consistent?” Reading all 200 files into one context would fill the window and the agent would lose the thread by file 60.

3a · Distill facts from a wide read

The answer to “too much to hold” is to carry the findings, not the files. As the agent traces a pattern across the codebase, have it extract the structured facts — which modules follow the pattern, which deviate, where the exceptions live — and carry those forward instead of the raw file contents. The context holds a growing map, not a growing pile.

3b · Delegate the crawl to a subagent

The cleaner move for a genuinely wide investigation: delegate it. A subagent reads the fifty relevant files in its own context and returns a concise summary — so the fifty files never land in your main agent’s window. The main agent asks the question and gets the answer, not the crawl. This is the same context-isolation benefit subagents give any multi-agent system, applied to legacy exploration.

options = ClaudeAgentOptions( cwd=repo, allowed_tools=["Read", "Glob", "Grep", "Agent"], # Agent = can delegate agents={ "explorer": { "description": "Reads many files to trace one pattern; returns a concise summary.", "prompt": "Trace the requested pattern across the codebase. Return findings only, not file contents.", "tools": ["Read", "Glob", "Grep"], }, }, )

3c · A scratchpad for the map

For a big legacy system, keep the growing understanding on disk — an architecture note the agent updates as it learns. State lives in the file; the context holds only what the current question needs. A fresh session (or a reset when context stops paying its way) can pick up the map instead of re-deriving it.

Exam lens The reliable theme: a wide read is a context problem, and the fixes are fact extraction (carry findings, not files), delegation (a subagent isolates the crawl), and external state (a scratchpad). Scenarios describe an agent degrading while reading a large system; the fix is never “a bigger window,” it’s one of these three.
Verify Subagent definition shape (the agents object, Agent in allowed_tools) is current but version-sensitive. Confirm against code.claude.com/docs/en/agent-sdk/subagents.
Stage 3 done when
  • The agent answers a cross-cutting question about a large system without stalling.
  • A subagent handles the wide crawl and returns a summary, keeping the main context lean.
  • A scratchpad file holds the accumulating architectural map.
Stage 04

Generate boilerplate safely

Agentic Architecture & Orchestration · 27%

Now the agent earns write access. Generating boilerplate — a new module, a test scaffold, a handler that matches the existing ones — is where a productivity agent pays off. But writing files is the first real risk, so you grant it deliberately and gate what it can touch.

“Create a new endpoint following the pattern of the existing ones.” The agent should write the new files — and only the new files, in the right place, matching your conventions — not reach into unrelated parts of the tree.

4a · Grant Write and Edit

Add the write tools to the list, and switch off pure plan mode so the agent can act. acceptEdits lets it write without prompting on every file — appropriate once you trust the task and are running in a safe environment.

options = ClaudeAgentOptions( cwd=repo, allowed_tools=["Read", "Glob", "Grep", "Write", "Edit"], permission_mode="acceptEdits", # write without per-file prompts can_use_tool=write_gate, # but still gate WHERE it writes (4b) max_turns=25, )

4b · Gate where it writes

Write access without scope is how an agent edits a file it shouldn’t. A can_use_tool gate keeps writes inside the intended area — deny anything outside the target directory, protect generated files or lockfiles, and let the rest through. The grant is broad (“can write”); the gate makes it precise (“can write here”).

async def write_gate(tool_name, args, context): if tool_name in ("Write", "Edit"): path = args.get("path", "") if not path.startswith("api/endpoints/"): return PermissionResultDeny(message="Only write under api/endpoints/ for this task.") return PermissionResultAllow()

4c · Match conventions by giving the agent the pattern

Good boilerplate follows the existing code. Because the agent can already read the repo (Stages 1–2), point it at an exemplar: “follow the pattern in api/endpoints/orders.py.” It reads the exemplar, matches structure and error handling, and generates the new files consistently — far better than generating from a blank prompt. Project configuration (Stage 7) makes those conventions automatic.

Exam lens Two ideas tested here. First, widen privilege deliberately — grant Write/Edit only when the task needs it, not up front. Second, a grant and a gate are different layers: allowed_tools says the agent can write; can_use_tool says where. Scenarios that ask “how do you let it generate code but not touch production config” want the gate, not a prompt.
Verify PermissionResultAllow/Deny, the can_use_tool signature, and acceptEdits auto-approval scope (in-working-directory rules) are current but version-sensitive. Confirm against code.claude.com/docs/en/agent-sdk/permissions.
Stage 4 done when
  • The agent generates new boilerplate that matches an existing exemplar’s pattern.
  • A gate confines its writes to the intended directory and denies anything outside.
  • You can explain the difference between granting a tool and gating where it acts.
Stage 05

Automate a repetitive task

Tool Design & MCP Integration · 18%

The last built-in capability, and the most powerful: Bash. Running commands is what turns the agent from a code generator into an automator — run the tests, apply a codemod across the tree, regenerate fixtures. It’s also the tool that can do the most damage, so it gets the most careful guard.

“Rename this function everywhere and update the imports, then run the test suite to confirm nothing broke.” That’s a real chore — and it’s exactly the kind of multi-step, verify-as-you-go work Bash unlocks.

5a · Grant Bash, in a sandbox

Add Bash to the list — but this is the point where the environment matters as much as the permission list. Run the agent in a container or VM, so even a mistaken command is contained. The sandbox is a safety layer the tool list can’t provide on its own.

options = ClaudeAgentOptions( cwd=repo, allowed_tools=["Read", "Glob", "Grep", "Edit", "Bash"], can_use_tool=bash_gate, # gate dangerous commands (5b) max_turns=25, )

5b · Gate dangerous commands

Not every shell command should run unquestioned. A gate can allow the safe, expected commands (the test runner, a formatter, git status) and deny or escalate the dangerous ones (anything that deletes broadly, touches credentials, or reaches the network). Scope the grant to intent, the same way you scoped writes in Stage 4.

# Allow the task's expected commands; deny the rest. SAFE = ("pytest", "ruff", "git status", "git diff") async def bash_gate(tool_name, args, context): if tool_name == "Bash": cmd = args.get("command", "") if not cmd.startswith(SAFE): return PermissionResultDeny(message="Command not in the allowed set for this task.") return PermissionResultAllow()
For a guarantee that must hold no matter what — never touch .env, never rm -rf — a PreToolUse hook is stronger than a callback: it runs first in the permission order and its deny holds even in permissive modes. Put hard rules there.

5c · Verify the automation worked

Automation you can’t verify isn’t safe to trust. The strength of an agent with Bash is that it can check its own work: after the rename, it runs the test suite; if tests fail, it reads the failure and fixes it; it re-runs to confirm. Build that verify-as-you-go loop into the task prompt, so “done” means “tests pass,” not “edits made.”

Exam lens Bash is the highest-capability, highest-risk built-in. Know that its safety comes from two layers: a gate/hook on which commands run, and a sandboxed environment around the whole agent. Know that hard guarantees belong in a PreToolUse hook (deny wins), and that verify-as-you-go (run tests, fix, re-run) is what makes automation trustworthy. “Trust the agent not to run bad commands” is the wrong answer.
Verify Bash gating, hook events (PreToolUse), and sandbox/permission-mode behavior move between versions, and the simple startswith check above is illustrative — use a robust matcher in practice. Confirm against the permissions and hooks docs.
Stage 5 done when
  • The agent completes a multi-step chore with Bash — edit, then run something to verify.
  • A gate (or hook) confines Bash to the task’s expected commands and blocks the rest.
  • The agent runs the tests, and “done” means they passed — it’s running in a sandbox.
Stage 06

Integrate an MCP server

Tool Design & MCP Integration · 18%

The built-in tools cover the filesystem, the shell, and the web. Everything else — your issue tracker, your database, your internal APIs — lives behind your own systems, and that’s what MCP is for. Connect one server and the agent can reach beyond the repo into the systems developers actually work against.

“Look at the open issues tagged bug, find the one about the login timeout, and show me the code responsible.” The issues live in a tracker, not the filesystem — so the agent needs a tool that reaches the tracker.

6a · Two kinds of MCP server

There are two ways to give the agent an MCP tool, and the exam expects you to know the difference:

  • External server — an existing server (a GitHub tracker, a database, a browser via Playwright). You connect to it by command/args (stdio) or a URL (HTTP/SSE). The agent connects as a client.
  • In-process server — your own domain logic, registered as functions inside your process with create_sdk_mcp_server. No subprocess, no network hop — for capabilities that live behind your API.

6b · Connect an external server

Pass the server config in mcp_servers and add its tools to allowed_tools. Tool names are fully qualified as mcp__<server>__<tool> — a common trip-up: that full string is what goes in the allow list.

options = ClaudeAgentOptions( cwd=repo, allowed_tools=["Read", "Glob", "Grep", "mcp__github__*"], # qualified names mcp_servers={ "github": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"], "env": {"GITHUB_PERSONAL_ACCESS_TOKEN": os.environ["GITHUB_TOKEN"]}, }, }, )
The MCP server config is a discriminated union — stdio (command/args), HTTP/SSE (url), and the in-process server are all valid shapes. Before relying on a specific one, list the connected server’s tools with a quick test so you know exactly what it exposes.

6c · Scope the MCP tools too

Least privilege applies to MCP just as it does to built-ins. Don’t grant mcp__github__* if the agent only needs to read issues — list only the specific tools it uses. An MCP server can expose write actions (close an issue, merge a PR); scope the allow list to exactly what the task requires, and gate the risky ones with can_use_tool.

Exam lens Know the two MCP shapes: external (connect by command/URL) vs. in-process (your own functions via create_sdk_mcp_server), and when each fits — in-process for your domain logic, external for existing servers. Know the mcp__server__tool naming goes in allowed_tools. And that least privilege extends to MCP tools — scope them, don’t wildcard everything.
Verify MCP server packages, the config union’s exact discriminant fields, and wildcard support in allowed_tools vary by version and server. Confirm the config shape and list the server’s tools before relying on specific names.
Stage 6 done when
  • The agent uses an MCP server to reach a system beyond the filesystem.
  • You can explain when to use an external server vs. an in-process one.
  • The MCP tools are scoped to what the task needs — no blanket wildcard on write actions.
Stage 07

Configure it for the team

Claude Code Configuration & Workflows · 20%

A productivity agent that only works on your machine helps one developer. The payoff is when the whole team gets the same agent, with the same conventions and the same guardrails. The SDK can load your project’s filesystem configuration, so the agent inherits what the team already agreed on.

7a · Load project configuration

The SDK doesn’t read your project’s CLAUDE.md and settings by default — you opt in. Set setting_sources=["project"] and the agent picks up the committed project context: conventions, build commands, path rules. Now “follow our conventions” is automatic, not something you re-specify each run.

options = ClaudeAgentOptions( cwd=repo, setting_sources=["project"], # load committed CLAUDE.md, rules, settings allowed_tools=["Read", "Glob", "Grep", "Edit"], )

7b · Encode the guardrails as shared config

The gates and hooks you built in Stages 4–5 shouldn’t live only in your script. Commit them as project settings — hooks in .claude/settings.json, path rules in .claude/rules/ — so every developer’s agent enforces the same guarantees. A guardrail that isn’t shared protects one person; committed, it protects the team.

7c · Package the agent for reuse

Wrap the whole thing — the options, the tool scoping, the gates, the loaded config — as a small CLI or internal tool your team runs. The build-vs-buy line from the SDK docs applies: an agent is worth it for open-ended work where you can’t hardcode the steps (explore this unfamiliar system, generate to spec); a plain script is better when the workflow is fixed.

Exam lens Know that the SDK loads filesystem config only when you opt in with setting_sources=["project"] — a common “why isn’t my CLAUDE.md loading” answer. Know that shared config (committed) reaches the team while personal settings don’t, and the agent-vs-script judgment: agents for open-ended work, scripts for fixed workflows.
Verify setting_sources (and its TypeScript form settingSources) and which config files it loads are current but evolving. Confirm against the SDK configuration docs.
Stage 7 done when
  • The agent loads the project’s committed config and follows team conventions automatically.
  • Your gates and guardrails are committed as shared settings, not just in your script.
  • The agent is packaged so a teammate can run it — and you can say why it’s an agent, not a script.
Wrap-up

What you built, and how it maps

You now have one working developer-productivity agent that exercises the scenario’s three primary domains: built-in tools scoped by least privilege plus MCP integration for reaching your own systems (Tool & MCP, 18%); the agent loop, deliberate privilege-widening, and gated write/Bash grants (Agentic Architecture, 27%); and context management for large legacy systems plus project configuration for team reuse (Context & Reliability, 15%, and Claude Code Config, 20%). Completing it, you should be able to answer any question the scenario raises — which tool fits which job, why allowed_tools is a security boundary, when to reach for MCP, and how to keep an agent coherent across a sprawling codebase.

The exam is 60 scenario questions in 120 minutes, and it rewards judgment over recall — the wrong answers are written to tempt people with partial knowledge. Having actually scoped the tools, gated the writes, and felt why least privilege is the right default is the difference between recognizing the right architectural call and guessing.
